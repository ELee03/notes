10 · Model-Based Reinforcement Learning
Dyna Architecture · World Models · Planning · Sample Efficiency
Every algorithm in Lessons 4–9 learned from experience directly — adjusting value estimates or policy parameters each time a real transition occurred. This works, but it's wasteful: each real interaction with the environment is expensive (time, physical wear, safety risk), yet is used for exactly one update and then discarded. Model-based RL asks: what if we also learned a model of the environment, so we could generate additional simulated experience for free? The answer is dramatically better sample efficiency — the same real-world data goes much further when combined with a learned model.
Notation recap
$\hat{p}(s'|s,a)$Learned transition model — predicted probability of next state
$\hat{r}(s,a)$Learned reward model — predicted reward
$n$Planning steps per real step in Dyna
$\mathcal{M}$The learned world model $(\hat{p}, \hat{r})$
Model-free vs model-based
Model-free RL (Lessons 6–9) learns a value function or policy directly from experience. No model of the environment is ever built — the agent just patterns-matches from $(s, a, r, s')$ tuples to estimates of $v_\pi$ or $q_*$.
Model-based RL learns an explicit model $\mathcal{M} = (\hat{p}, \hat{r})$ from experience, then uses that model to plan — generating simulated transitions to improve the policy without additional real interaction.
The central tradeoff:
Model-free Model-based
Sample efficiencyLow — each real step used onceHigh — real steps reused via simulation
Computation per stepLowHigher (model learning + planning)
Model errorNone (no model)Compound — errors in $\hat{p}$ compound over planning steps
Best forCheap simulation, video gamesReal robots, expensive interactions
The Dyna architecture
Dyna (Sutton 1991) is the canonical model-based RL framework. The idea is simple: after each real interaction, do two things — (1) update the model with the new experience, and (2) run $n$ simulated updates using the model to refine the value function. The value function update rule is exactly Q-learning or SARSA — the only difference is whether the $(s, a, r, s')$ tuple came from the real environment or the model.
Dyna-Q algorithm
  1. Observe $S_t$, choose $A_t$ (ε-greedy from $Q$).
  2. Take action, observe $R_{t+1}$, $S_{t+1}$. (real step)
  3. Q-learning update: $Q(S_t, A_t) \leftarrow Q(S_t, A_t) + \alpha[R_{t+1} + \gamma\max_{a'}Q(S_{t+1},a') - Q(S_t,A_t)]$
  4. Model update: store $(S_t, A_t) \to (R_{t+1}, S_{t+1})$ in $\mathcal{M}$.
  5. Planning loop ($n$ times): sample a previously observed $(s, a)$ from memory, query $\mathcal{M}$ for $(\hat{r}, \hat{s}')$, run a Q-learning update on the simulated tuple.
Each real step is worth $n+1$ Q-learning updates: one from reality, $n$ from the model. With a good model, this $n$-fold amplification translates directly into faster learning with fewer real interactions. In Sutton's original maze experiments, Dyna with $n=50$ solved mazes in roughly 1/50th the real interactions of pure Q-learning.
What counts as a "model"? In simple tabular settings, the model is just a lookup table: every $(s, a)$ seen so far maps to its empirically observed $(r, s')$. For continuous state spaces, the model is a function approximator (neural network, Gaussian process, or linear model) trained to predict $(\hat{r}, \hat{s}')$ from $(s, a)$.
Planning with a learned model
Planning is the process of using a model to improve a policy without real interaction. The key insight from DP (Lesson 5): once you have a model, you can apply Bellman updates to any state — not just the ones you've recently visited. This is what makes model-based methods so sample-efficient.
Several planning strategies are used in practice:
  • Random sampling (Dyna). Sample $(s, a)$ uniformly at random from all previously visited pairs, query the model, update $Q$. Simple and effective.
  • Prioritized sweeping. Prioritize updates by the magnitude of the expected TD error — states whose values have changed significantly (or are expected to change) get updated first. Propagates value information more efficiently backward from recently discovered rewards.
  • Trajectory sampling. Simulate entire rollouts forward from the current state under the current policy. Useful when the value of nearby states is more relevant than random past states.
  • Background planning (MCTS). Monte Carlo Tree Search builds a search tree by running many short simulations from the current state, choosing which branches to expand based on visit counts and estimated values. Used in AlphaGo/AlphaZero.
World models
In deep RL, the learned model $\mathcal{M}$ is called a world model. It typically consists of:
  • Representation model — encodes high-dimensional observations (images, sensor arrays) into a compact latent state $z_t$. Often a convolutional encoder.
  • Transition model — predicts the next latent state: $\hat{z}_{t+1} = f(z_t, a_t)$. This is where "model rollouts" happen — rolling the transition model forward $k$ steps to simulate future states.
  • Reward model — predicts reward from latent state and action: $\hat{r}_t = g(z_t, a_t)$.
Planning then happens entirely in the latent space: the policy sees $z_t$ rather than raw observations, and model rollouts generate $(z_t, a_t, \hat{r}_t, z_{t+1})$ tuples in latent space, which are used to update the value function and policy.
Dreamer (Hafner et al. 2019–2023) is the leading world model architecture. It learns a recurrent world model in latent space and trains the policy entirely on model-imagined trajectories — the policy never sees real environment transitions during policy gradient updates. On continuous control benchmarks, Dreamer achieves with $10\times$–$50\times$ fewer real environment steps than model-free baselines.
The model error problem
The fundamental challenge of model-based RL is compounding model error. A model trained on data from the current policy will be accurate near states it has seen, but less accurate in unexplored regions. If the planner rolls out the model for many steps, small per-step errors compound geometrically — by step $k$, the simulated state may have diverged significantly from reality.
This creates a distribution shift problem: the policy is being trained on simulated data from states that the model misrepresents. A policy trained on wrong model outputs can be worse than no model at all. The failure mode is model exploitation: the planner finds action sequences that look good in the model but fail in reality, because the model has systematic errors in exactly those regions.
Mitigation strategies:
  • Short rollouts. Limit the planning horizon to $k$ steps (e.g. 1–5). Error compounds less over short horizons. Dyna uses $k=1$: each simulated update is a single one-step model prediction.
  • Ensemble models. Train an ensemble of $M$ models; use variance across models to detect high-uncertainty regions and avoid planning there.
  • Model predictive control (MPC). Replan at every step: roll the model out $k$ steps to select the best action, take one action, observe the real outcome, replan. Never trust the model for more than one step at a time.
  • Conservative planning. Penalise the planner for visiting states where the model is uncertain (equivalent to adding a pessimism bonus to the cost function).
Sample efficiency: model-free vs model-based
The sample efficiency gap between model-based and model-free is striking. On continuous control benchmarks (e.g. MuJoCo locomotion tasks):
  • SAC (model-free): typically requires ~$10^6$ environment steps to solve a standard locomotion task.
  • Dreamer (model-based): achieves comparable performance in ~$10^5$ steps — 10× fewer.
  • MBPO (model-based, short rollouts): ~$5\times10^4$ steps on some tasks — another 2× improvement.
This gap matters enormously for real-world applications. A physical robot arm executing $10^6$ real-world trials would take months. $10^4$–$10^5$ trials are feasible within hours to days. This is why model-based methods dominate robotics applications.
When not to use a model. Model-based methods have higher computational overhead and can fail badly when the environment is hard to model (stochastic rewards, partial observability, highly chaotic dynamics). For fast simulators (Atari games, physics engines), model-free methods with enough data often win — the simulation is already the "free" experience source, so sample efficiency doesn't matter.
Worked examples
Example 1 — Dyna-Q update trace
A simple 2-state MDP: states $\{A, B\}$, actions $\{L, R\}$, $\gamma = 0.9$, $\alpha = 0.1$. From state $A$, action $R$ always leads to $B$ with reward $+1$. All other transitions stay in the same state with reward $0$. $Q$ is initialized to 0 everywhere. After two real steps $(A, R, +1, B)$ and $(B, L, 0, B)$, run $n=2$ planning steps. What are $Q(A,R)$ and $Q(B,L)$ after all updates?
Real step 1: $(A, R, +1, B)$. $Q(A,R) \leftarrow 0 + 0.1[1 + 0.9\cdot\max(Q(B,\cdot)) - 0] = 0.1[1 + 0] = 0.1$ Model stores: $(A,R) \to (+1, B)$. Real step 2: $(B, L, 0, B)$. $Q(B,L) \leftarrow 0 + 0.1[0 + 0.9\cdot\max(Q(B,\cdot)) - 0] = 0$ Model stores: $(B,L) \to (0, B)$. Planning step 1: sample $(A,R)$ from model. $\hat{r}=1$, $\hat{s}'=B$. $Q(A,R) \leftarrow 0.1 + 0.1[1 + 0.9\cdot 0 - 0.1] = 0.1 + 0.09 = 0.19$ Planning step 2: sample $(B,L)$ from model. $\hat{r}=0$, $\hat{s}'=B$. $Q(B,L) \leftarrow 0 + 0.1[0 + 0.9\cdot 0 - 0] = 0$ (unchanged). Final: $Q(A,R) = 0.19$, $Q(B,L) = 0$. The model amplified the real-step update to $Q(A,R)$ without an additional real interaction.
Example 2 — Compounding model error
A 1D system: $s_{t+1} = s_t + a_t + \varepsilon$ where $\varepsilon \sim \mathcal{N}(0, 0.01)$ is environment noise. The learned model predicts $\hat{s}_{t+1} = s_t + a_t + 0.1$ (systematic bias of $+0.1$). If we plan $k=10$ steps into the future with action $a=0$, what is the expected model prediction error vs reality?
Under the real environment (no noise): $s_{t+k} = s_t$. Under the model: $\hat{s}_{t+k} = s_t + k \cdot 0.1 = s_t + 1.0$ after 10 steps. The per-step bias of $0.1$ compounds linearly over the horizon: after 10 steps, the simulated state is 1.0 units off from reality — even though the single-step model error is only $0.1$. This illustrates why model rollout lengths are kept short in practice (typically 1–5 steps for continuous control). The usable planning horizon is roughly $1/\varepsilon_{\text{model}}$ steps before error becomes too large.
Food for thought
Reflection questions
  1. Dyna uses random sampling from the model replay buffer — picking any previously seen $(s,a)$ pair uniformly. Prioritized sweeping instead focuses updates on states with large expected TD errors. In what environment structure would prioritized sweeping dramatically outperform random Dyna? When might random sampling do just as well?
    States far from a newly discovered reward signal gain large TD errors first. In a long maze where the reward is at the far end, prioritized sweeping propagates value information backward efficiently — each update triggers a cascade of high-error predecessors. In a small or densely connected MDP where most states are equally likely to be informative, random sampling is fine.
  2. A world model trained on data from a random policy will be accurate near random trajectories but unreliable in states a trained policy would visit. Why does this create a "chicken and egg" problem, and how do active learning or exploration bonuses help?
    To get a good model, you need good coverage of the state space — but to get good coverage, you need a policy that explores. A random policy covers states randomly (often missing task-relevant states); a trained policy concentrates on states it already knows. Exploration bonuses (e.g. count-based or curiosity-driven exploration) reward visiting novel states, incentivising the policy to collect data in under-explored regions where the model is inaccurate — breaking the cycle.
  3. AlphaZero uses MCTS planning with a learned model (the neural network provides prior probabilities and value estimates). What property of board games makes deep planning (many-step lookahead) reliable there, whereas in continuous control it's usually limited to 1–5 steps?
    Board games are deterministic and fully observable — the model is exact (the rules of chess are known). There is no model error to compound. Planning 20–30 moves ahead produces perfectly accurate future states. In continuous control, even tiny model errors compound geometrically, making long rollouts unreliable. The useful planning horizon is determined by the per-step model error, not by the algorithm's capacity to plan.
Common pitfalls
Mistakes worth knowing about in advance
  • Trusting the model for long rollouts. Even a model with 99% per-step accuracy becomes $0.99^{50} \approx 60\%$ accurate after 50 steps. Keep rollout horizons short and compensate with more model updates.
  • Forgetting to update the model as the policy changes. The model was trained on data from the old policy. As the policy improves and visits new states, the model becomes stale in those regions. Continuously retraining (or using an experience replay buffer with recent samples) is essential.
  • Using model error as a reward signal (naive curiosity). If you reward the agent for visiting states the model predicts poorly, you incentivise it to seek out physically unrealizable states that confuse the model — the "noisy TV problem." Curiosity-driven exploration requires careful design.
  • Conflating planning with policy gradient. In model-based policy gradient methods (e.g. MBPO), model rollouts are used to augment the real data buffer. The policy is still updated by SAC or PPO on the augmented buffer — the model generates data, not gradients. Don't confuse this with differentiable model-based methods (e.g. PILCO) where gradients flow back through the model.
