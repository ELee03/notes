5 · Dynamic Programming
Policy Evaluation · Policy Improvement · Policy Iteration · Value Iteration · Known Model
The Bellman equations from Lesson 4 are a characterisation — they tell you what the value function must look like, but not how to compute it. Dynamic programming turns them into algorithms. The key assumption here is that you have a complete model of the environment: you know $p(s',r \mid s,a)$ for every state, action, and outcome. Given that, DP computes value functions exactly through repeated sweeps over the state space, each pass bringing the estimate closer to the truth. Lessons 6 and 7 will drop this model requirement — but DP is where the algorithmic ideas first take shape.

The dynamic programming idea
The word "dynamic" refers to sequences of decisions over time; "programming" here means optimisation (in the 1950s sense, not software). Richard Bellman coined the term to describe the principle that optimal decisions at each stage can be found by working backward through the problem — each stage's optimal choice depends only on the value of what follows.
In the RL context, DP exploits the fact that $v_\pi$ satisfies a self-consistency condition (the Bellman equation). If you start with any initial estimate $V_0$ — even all zeros — and repeatedly apply the Bellman update, the estimates converge to the true $v_\pi$. This works because the Bellman operator is a contraction mapping in the max-norm: each application reduces the error by a factor of $\gamma$. After enough sweeps, $V_k \to v_\pi$.
Policy evaluation (the prediction problem)
Given a policy $\pi$, compute $v_\pi$. The algorithm maintains an array $V$ of value estimates for each state and iteratively updates them using the Bellman expectation equation:
$$V_{k+1}(s) \;\leftarrow\; \sum_a \pi(a \mid s) \sum_{s',\, r} p(s', r \mid s, a) \;\bigl[r + \gamma V_k(s')\bigr] \qquad \text{for all } s \in \mathcal{S}$$
Each pass over all states is called a sweep. After each sweep, the maximum change across all states decreases. Stop when $\max_s |V_{k+1}(s) - V_k(s)| < \theta$ for some small threshold $\theta$.
Why does this converge? The Bellman expectation operator $\mathcal{T}^\pi V(s) = \sum_a \pi(a|s) \sum_{s',r} p(s',r|s,a)[r + \gamma V(s')]$ is a $\gamma$-contraction in the infinity norm: $\|\mathcal{T}^\pi V - \mathcal{T}^\pi V'\|_\infty \leq \gamma \|V - V'\|_\infty$. By the Banach fixed-point theorem, repeated application of any contraction converges to a unique fixed point — which is exactly $v_\pi$. The factor $\gamma < 1$ is what guarantees convergence.
Policy improvement
Once you have an estimate of $v_\pi$, you can ask: is there a better policy? The greedy policy $\pi'$ with respect to $v_\pi$ selects, in each state, the action that maximises the one-step look-ahead value:
$$\pi'(s) = \arg\max_a \sum_{s',\, r} p(s', r \mid s, a)\;\bigl[r + \gamma v_\pi(s')\bigr] = \arg\max_a\; q_\pi(s, a)$$
The Policy Improvement Theorem guarantees that $\pi'$ is at least as good as $\pi$: for all $s$, $v_{\pi'}(s) \geq v_\pi(s)$. The intuition is clean — if $\pi'$ is choosing the action with the highest $q_\pi$ value while $\pi$ might not, then following $\pi'$ (greedily for one step, then $\pi$ after) must be at least as good as following $\pi$ all the time.
⟶ Worth working through the Policy Improvement Theorem yourself. Start from $q_\pi(s, \pi'(s)) \geq q_\pi(s, \pi(s)) = v_\pi(s)$ and expand $q_\pi$ recursively to show $v_{\pi'}(s) \geq v_\pi(s)$. The key step uses the fact that following $\pi'$ at the first step is an improvement, and that the same argument applies at every subsequent step. This is the prototype for all policy improvement arguments in RL.
If $\pi' = \pi$ (no action changed), then we have hit a fixed point of the improvement step: $v_\pi = v_*$, and $\pi$ is already optimal. Policy improvement either strictly improves the policy or confirms it is already optimal.
Policy iteration
Alternate between evaluation and improvement until convergence:
Policy iteration converges in a finite number of steps. There are finitely many deterministic policies ($|\mathcal{A}|^{|\mathcal{S}|}$), and each iteration either strictly improves the policy or terminates. In practice, policy iteration often converges in fewer iterations than you might expect — the value function does much of the work in a few sweeps.
Value iteration
Policy iteration requires a full evaluation of $v_\pi$ (many sweeps to convergence) before each improvement step. This is expensive. Value iteration combines them: instead of evaluating $v_\pi$ exactly, apply just one sweep of the Bellman optimality update:
$$V_{k+1}(s) \;\leftarrow\; \max_a \sum_{s',\, r} p(s', r \mid s, a) \;\bigl[r + \gamma V_k(s')\bigr] \qquad \text{for all } s \in \mathcal{S}$$
This is the Bellman optimality operator applied repeatedly. It is still a contraction (the max-norm argument holds for the optimality operator too), so $V_k \to v_*$. Once $V$ has converged, extract the greedy policy: $\pi^*(s) = \arg\max_a \sum_{s',r} p(s',r \mid s,a)[r + \gamma V(s')]$.
Conceptually: value iteration is doing policy evaluation (one sweep only) and policy improvement simultaneously. Each sweep improves both the value estimate and the implicit greedy policy. It is faster per iteration than policy iteration but typically needs more iterations to converge.

Limitations of dynamic programming
DP is exact and computationally well-understood, but it has two fundamental limitations that motivate everything in Lessons 6–9:
  • Requires a complete model. The update rule $V_{k+1}(s) = \max_a \sum_{s',r} p(s',r|s,a)[\ldots]$ uses $p(s',r|s,a)$ explicitly. For most real problems — robots in physical spaces, games with complex rules, financial systems — this distribution is unknown. You can sample from the environment but not enumerate it.
  • Visits every state every sweep. DP iterates over the entire state space on each sweep. For practical problems with large or continuous state spaces, this is computationally infeasible. Asynchronous DP variants (updating only some states per sweep) partially address this, but the fundamental issue remains.
These limitations are not fatal — they are boundaries that define where DP works and where you need something else. Monte Carlo methods (Lesson 6) address the model requirement by learning from sampled episodes. Temporal Difference (Lesson 7) addresses both requirements by bootstrapping from partial trajectories. Function approximation (Lesson 8) addresses the large-state-space problem.
Food for thought
Reflection questions — no answers given
  1. Policy iteration uses exact evaluation before each improvement. But you only need an estimate good enough to identify which action is best — not exact values. How much work do you think you're wasting, and what would you do differently? (This question is essentially asking you to re-invent modified policy iteration — look it up after thinking it through.)
  2. In value iteration, a single state can influence all its neighbours in one sweep, but those neighbours haven't yet "seen" their new values from other directions. How does the information about terminal states propagate outward with each sweep? How many sweeps does a state $k$ steps from a terminal need before its value first becomes nonzero?
  3. DP assumes the model is known. But what if you knew the model approximately — say, you had samples from $p(s',r|s,a)$ but not the exact distribution? Could you still run something like policy evaluation? What would you lose?
Common pitfalls
Mistakes worth knowing about in advance
  • Confusing a sweep with convergence. One sweep over all states is not evaluation — it is one step toward evaluation. After a single sweep, $V_1$ still has significant error. Convergence requires many sweeps; the exact number depends on $\gamma$ and the MDP structure. Stopping too early gives poor values and a poor improved policy.
  • Using in-place vs. two-array updates interchangeably. You can update $V$ in-place (using new values as soon as they are computed within a sweep) or with a separate copy. Both converge, but in-place typically converges faster because newly computed values propagate immediately. They are not equivalent; pick one and be consistent.
  • Thinking policy improvement always gives a strictly better policy. Policy improvement either strictly improves the policy or confirms it is already optimal. If the new greedy policy matches the old policy everywhere, you have reached $v_*$ — not stagnation. This is the stopping criterion for policy iteration.
  • Forgetting that value iteration needs a final greedy pass. Value iteration converges the value function $V \to v_*$, but the algorithm itself does not explicitly maintain a policy during iteration. You must extract the optimal policy at the end with one greedy pass: $\pi^*(s) = \arg\max_a[\ldots]$. Stopping value iteration early and reading off the greedy policy from an unconverged $V$ gives a suboptimal policy.
Connects to
  • Linear Algebra — the policy evaluation linear system $v = r^\pi + \gamma P^\pi v$ is solved iteratively here; its direct solution is $(I - \gamma P^\pi)^{-1} r^\pi$. The spectral radius of $\gamma P^\pi$ being less than 1 guarantees the Neumann series converges — this is the convergence proof for policy evaluation in matrix form.
  • RL · Monte Carlo Methods (Lesson 6) — MC drops the model requirement of DP by estimating value functions from complete sampled episodes. The same policy evaluation target ($v_\pi$) is estimated, but by averaging sample returns rather than by solving the Bellman equations.
  • Control Theory · LQR (Lesson 13) — the algebraic Riccati equation in LQR is the continuous-time analog of the Bellman optimality equation here. Value iteration for LQR is the backward Riccati recursion.
