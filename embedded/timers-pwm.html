6 · Timers & PWM
Counter Modes · Compare/Capture · PWM · Dead-Time · LETIMER
A hardware timer is a counter clocked from the peripheral bus (or a prescaled fraction of it) that can trigger events when it reaches programmed values, generate precisely timed output waveforms, and latch the current count when an external edge arrives. Timers are used everywhere in embedded systems: scheduling periodic tasks, generating PWM for motors and LEDs, measuring pulse widths from sensors, creating audio tones, and synchronising ADC acquisition to a sample clock.
The EFR32 family has several timer peripherals: TIMER0–TIMER3 (16-bit, full-featured, clocked from HFPERCLK), WTIMER0–WTIMER1 (32-bit versions on some variants), and LETIMER0 (16-bit, low-energy, clocked from the 32 kHz LFXO/LFRCO and able to run in EM2). The full-featured TIMERs are the right choice for PWM and fast measurement; LETIMER is the right choice for periodic low-power wakeup.
Counter modes
At its heart a timer is a counter register CNT that increments (or decrements) on each tick of the timer clock. The TOP register sets the ceiling. When CNT reaches TOP an overflow event fires, an interrupt can be raised, and the counter resets.

Compare/Capture channels
Full-featured timers expose several Compare/Capture (CC) channels — each is an independent sub-unit with its own value register and output pin. The same hardware block does two different things depending on the mode:
  • Output compare (OC): the CNT register is compared to the CC value every tick. On a match, the channel performs a configurable action — toggle the output pin, set it high, set it low, or generate an interrupt. Used for PWM output and precise timing events.
  • Input capture (IC): the CC pin is an input. On a configured edge (rising, falling, or both), the current CNT value is copied into the CC value register and an interrupt fires. Used to measure pulse widths, periods, and phase differences.
PWM generation
Pulse-Width Modulation (PWM) uses a fixed-frequency switching waveform with a variable duty cycle to encode an analogue value in a digital signal. Average power into a load — an LED, a motor winding, a speaker coil — is proportional to duty cycle. At high enough switching frequency the inductance or persistence-of-vision of the load integrates the PWM into a smooth average.
The output is HIGH while CNT < CC (compare value), LOW while CNT ≥ CC. Duty cycle = CC / (TOP + 1).
The formula for PWM frequency from an up-counting timer:
fPWM = fPCLK / (prescaler × (TOP + 1))     duty = CC / (TOP + 1) × 100%
Choosing TOP for a target frequency. Rearrange: TOP = f_PCLK / (prescaler × f_PWM) − 1. At 38.4 MHz with prescale=1, a 10 kHz PWM requires TOP = 38400000/10000 − 1 = 3839. Resolution is then log₂(3840) ≈ 11.9 bits. Lowering f_PWM improves resolution; raising it reduces it. For 16-bit timers, the hard limit is TOP ≤ 65535. Use a prescaler to bring very low frequencies into range without sacrificing resolution.
Input capture — measuring time
Input capture is the timer in reverse: instead of producing events at programmed counter values, it records the counter value at the moment an external edge arrives. Two captures on successive edges of the same signal give you period or pulse width with the resolution of the timer clock. 
// Period measurement using CC0 in capture mode
// Capture on every rising edge; period = difference between successive captures

static volatile uint32_t last_capture = 0;
static volatile uint32_t period_ticks = 0;

void TIMER0_CC0_IRQHandler(void) {
    uint32_t now = TIMER_CaptureGet(TIMER0, 0);
    TIMER_IntClear(TIMER0, TIMER_IF_CC0);

    if (last_capture > 0) {
        // Handle 16-bit overflow if last > now
        period_ticks = (now >= last_capture)
                     ? (now - last_capture)
                     : (0xFFFF - last_capture + now + 1);
    }
    last_capture = now;
}

// Period in seconds = period_ticks / f_timer
// Frequency = f_timer / period_ticks
EFR32 TIMER API
The EMLIB TIMER driver centers on two init structures: TIMER_Init_TypeDef for the counter itself (mode, prescaler, enable-on-init) and TIMER_InitCC_TypeDef for each CC channel (compare/capture/PWM mode, output polarity, edge sensitivity). 
#include "em_timer.h"
#include "em_cmu.h"
#include "em_gpio.h"

#define PWM_FREQ_HZ  10000ul   // 10 kHz

void timer_pwm_init(void) {
    CMU_ClockEnable(cmuClock_TIMER0, true);
    CMU_ClockEnable(cmuClock_GPIO,   true);

    // ── 1. Configure the counter ───────────────────────────────
    TIMER_Init_TypeDef init = TIMER_INIT_DEFAULT;
    init.prescale = timerPrescale1;    // divide by 1 (full 38.4 MHz)
    init.mode     = timerModeUp;
    init.enable   = false;             // don't start yet
    TIMER_Init(TIMER0, &init);

    // Set TOP for the desired PWM frequency
    uint32_t f_timer = CMU_ClockFreqGet(cmuClock_TIMER0);
    uint32_t top = (f_timer / PWM_FREQ_HZ) - 1;  // e.g. 3839 for 10 kHz
    TIMER_TopSet(TIMER0, top);

    // ── 2. Configure CC0 for PWM output ────────────────────────
    TIMER_InitCC_TypeDef cc = TIMER_INITCC_DEFAULT;
    cc.mode = timerCCModePWM;
    cc.cmoa = timerOutputActionNone;   // not used in PWM mode
    TIMER_InitCC(TIMER0, 0, &cc);

    // Start at 50% duty cycle
    TIMER_CompareSet(TIMER0, 0, top / 2);

    // ── 3. Route CC0 output to a GPIO pin ─────────────────────
    GPIO_PinModeSet(gpioPortA, 0, gpioModePushPull, 0);
    TIMER0->ROUTELOC0 = TIMER_ROUTELOC0_CC0LOC_LOC0;
    TIMER0->ROUTEPEN  = TIMER_ROUTEPEN_CC0PEN;

    // ── 4. Start the timer ─────────────────────────────────────
    TIMER_Enable(TIMER0, true);
}

// Adjust duty cycle at runtime (0–100%)
void pwm_set_duty_pct(uint8_t pct) {
    uint32_t top = TIMER_TopGet(TIMER0);
    // Clamp to valid range
    if (pct > 100) pct = 100;
    TIMER_CompareSet(TIMER0, 0, (uint32_t)(top + 1) * pct / 100);
}
Series 2 routing. On EFR32 Series 2, timer output routing uses the GPIO peripheral's route registers instead of TIMER→ROUTELOC0:

GPIO->TIMERROUTE[0].CC0ROUTE = (gpioPortA << 16) | 0;
GPIO->TIMERROUTE[0].ROUTEEN = GPIO_TIMER_ROUTEEN_CC0PEN;
Dead-time insertion
Motor drives and switching power supplies use complementary pairs of MOSFETs — a high-side switch and a low-side switch in a half-bridge. If both switches are on at the same time, the supply rail is shorted to GND: this is called shoot-through, and it destroys FETs immediately. Even with theoretically complementary gate signals, real FETs have finite turn-on and turn-off delays, so you must insert a dead time — a short window where both switches are off — during every switching transition.
Red windows: both high-side and low-side off simultaneously. Prevents shoot-through during switching transitions.
The EFR32 TIMER has a built-in Dead-Time Insertion (DTI) unit that automatically generates the complementary output (CDTI) with configurable dead-time rise and fall periods. You do not need to manage the dead time in software. 
// ── Dead-Time Insertion for half-bridge motor drive ──────────
TIMER_InitDTI_TypeDef dti = TIMER_INITDTI_DEFAULT;
dti.enable              = true;
dti.activeLowOut        = false;   // active-high outputs
dti.invertComplementaryOut = true; // CDTI = NOT CC by default; DTI adds the gap
dti.prescale            = timerDtiPrescale1;
dti.riseTime            = 4;       // dead-time ticks before LS turns on
dti.fallTime            = 4;       // dead-time ticks before HS turns on
// At 38.4 MHz: 4 ticks = 104 ns dead time
TIMER_InitDTI(TIMER0, &dti);

// Enable CDTI0 output pin
GPIO_PinModeSet(gpioPortA, 1, gpioModePushPull, 0); // CDTI0 pin
TIMER0->ROUTEPEN  |= TIMER_ROUTEPEN_CDTI0PEN;
TIMER0->ROUTELOC2  = TIMER_ROUTELOC2_CDTI0LOC_LOC0;
Dead time must exceed FET turn-off time. Check the FET datasheet for toff (turn-off delay + fall time). The dead-time setting must be at least this value plus margin. Too little dead time → shoot-through; too much dead time → distorted output waveform (matters more for audio than motor control). Typical power FETs need 50–200 ns; SiC/GaN devices can be <20 ns.
LETIMER — low-energy periodic timer
The full-featured TIMERs run from HFPERCLK, which requires EM0 (active mode). For periodic tasks in a battery-powered device — a sensor poll every 100 ms, a heartbeat LED blink every second, a watchdog kick every 500 ms — you want a timer that runs in EM2 (deep sleep). That is LETIMER0.
LETIMER runs from the 32.768 kHz LFXO (or the ULFRCO at ~1 kHz for even lower power). It is 16-bit and counts down from COMP0 to 0, then reloads — generating an interrupt at each underflow. Its only job is to wake the CPU at regular intervals. 
#include "em_letimer.h"
#include "em_cmu.h"

#define WAKEUP_HZ  10  // 10 Hz = 100 ms period

void letimer_init(void) {
    // LETIMER runs from LFXO (must be enabled in CMU)
    CMU_OscillatorEnable(cmuOsc_LFXO, true, true);
    CMU_ClockSelectSet(cmuClock_LFA, cmuSelect_LFXO);
    CMU_ClockEnable(cmuClock_LETIMER0, true);

    LETIMER_Init_TypeDef init = LETIMER_INIT_DEFAULT;
    init.enable   = false;
    init.comp0Top = true;   // counter reloads from COMP0 on underflow
    LETIMER_Init(LETIMER0, &init);

    // Period = LFXO / WAKEUP_HZ = 32768 / 10 = 3277 ticks
    uint32_t comp0 = CMU_ClockFreqGet(cmuClock_LETIMER0) / WAKEUP_HZ;
    LETIMER_CompareSet(LETIMER0, 0, comp0);

    // Interrupt on COMP0 match (counts down to 0, reloads comp0)
    LETIMER_IntEnable(LETIMER0, LETIMER_IF_COMP0);
    NVIC_ClearPendingIRQ(LETIMER0_IRQn);
    NVIC_EnableIRQ(LETIMER0_IRQn);

    LETIMER_Enable(LETIMER0, true);
}

void LETIMER0_IRQHandler(void) {
    LETIMER_IntClear(LETIMER0, LETIMER_IF_COMP0);
    // Runs every 100 ms, even from EM2 deep sleep
    periodic_task();
}
The typical pattern for a BLE sensor device: the system enters EM2 in the main loop. LETIMER fires every N milliseconds, the NVIC wakes the CPU back to EM0 for the IRQ handler, the peripheral task runs (read sensor, update BLE advertisement), and the CPU returns to EM2. Total active time might be <1 ms per 100 ms period — 99% duty cycle in EM2.
