9 · SPI
Shift Register · CPOL/CPHA Modes · Chip Select · Multi-Device · EFR32 USART
SPI — Serial Peripheral Interface — is the workhorse of short-range, high-speed peripheral communication. Where UART is asynchronous and I2C is slow, SPI is synchronous and fast: a shared clock line means no baud rate negotiation, and full-duplex operation means master and slave exchange data simultaneously on every clock edge. Flash memory, display controllers, ADCs, DACs, IMUs, and RF transceivers almost all offer an SPI interface.
The trade-off is wires. SPI uses at least four lines per device — clock (SCK), master-out/slave-in (MOSI), master-in/slave-out (MISO), and chip select (CS/NSS) — plus one dedicated CS line per additional device on the bus. For a single peripheral this is fine; for ten peripherals it can become a pin budget problem.
Shift register operation
The core mechanism is a pair of shift registers — one in the master, one in the slave — connected in a loop. On each clock pulse one bit from the master's register shifts out onto MOSI and into the slave's register, while simultaneously one bit from the slave's register shifts out onto MISO and into the master's register. After N clock pulses the registers have swapped contents. This is full-duplex by construction: you always receive while you transmit.
SPI forms a circular shift register: MOSI carries master bits to the slave; MISO carries slave bits back. Both shift on the same clock, so every transfer is inherently full-duplex.
A consequence of the loop: even if you only want to receive (read a sensor register), you must clock out something on MOSI — the hardware does it anyway. Typically you send 0x00 or 0xFF as a dummy byte while reading. Similarly, a write-only transaction still shifts in garbage on MISO, which the firmware simply discards.
CPOL/CPHA clock modes
Unlike UART where the framing is standardised, SPI has no single agreed clock convention. Two parameters configure the relationship between the clock and data:
  • CPOL (Clock Polarity) — the idle state of SCK when no transfer is in progress. CPOL=0: idle LOW. CPOL=1: idle HIGH.
  • CPHA (Clock Phase) — which clock edge is used to sample data. CPHA=0: sample on the first (leading) edge. CPHA=1: sample on the second (trailing) edge.
These two bits give four modes, conventionally numbered 0–3:
Mode CPOL CPHA SCK idle Sample on Common devices
0 0 0 LOW Rising edge Most ADCs, SD cards, many sensors
1 0 1 LOW Falling edge Some IMUs (MPU-6000), MAX7219
2 1 0 HIGH Falling edge Some flash memories
3 1 1 HIGH Rising edge Some DACs, MCP2515 CAN controller
All four SPI clock modes. The red circles mark the sample point for each. Modes 0 and 3 both sample on rising edges but differ in idle polarity — they are not interchangeable.
The practical question when bringing up a new SPI device is: which mode does it use? The answer is always in the device datasheet, usually in the "SPI interface" or "timing diagram" section. Look for the SCK idle level and which edge is labelled "data valid" or "latch". If you misconfigure the mode, you will receive data shifted by one bit, or complete garbage.
Modes 0 and 3 are often confused. Both sample on the rising edge of SCK — but in Mode 0 the clock idles LOW (so the first edge is a rising edge), while in Mode 3 the clock idles HIGH (so the first edge is falling, and the rising edge is the second). Swapping them shifts all received bytes by one bit position.
Chip select management
Chip select (CS, also called NSS — Slave Select — or SS) is active-LOW: pulling it LOW wakes the device and tells it to participate in the current transfer. During the transfer, CS must stay LOW continuously; any glitch will cause the slave to reset its bit counter and lose synchronisation. CS goes HIGH between transfers to deselect the device.
There are two ways to drive CS from the master MCU:
  • Hardware NSS — the USART/SPI peripheral drives CS automatically: it asserts CS before the first bit and deasserts after the last bit. This is convenient but inflexible. Many SPI devices require CS to stay LOW across a multi-byte command (e.g. a register address followed immediately by data with no gap). Hardware NSS often deasserts CS after every byte, which breaks these devices.
  • Software CS — a regular GPIO pin, driven manually in firmware. You assert it (GPIO LOW), perform the full transaction (all bytes), then deassert it (GPIO HIGH). This gives complete control and is the common choice for real firmware. 
// Software CS pattern (the normal way)
GPIO_PinOutClear(CS_PORT, CS_PIN);      // assert CS (LOW)

USART_Tx(USART0, reg_addr | 0x80);     // send command byte (read bit set)
uint8_t result = USART_Rx(USART0);     // shift in response byte simultaneously

GPIO_PinOutSet(CS_PORT, CS_PIN);        // deassert CS (HIGH)
The CS pin itself should be configured as a push-pull output with a high initial value (deasserted) so it does not accidentally select the device at power-up or before the SPI peripheral is initialised.
Multi-device bus topologies
SPI does not have addressing. The master selects which device it is talking to purely by which CS line it asserts. This gives two practical topologies:
Left: independent CS — one GPIO per slave, all share SCK/MOSI/MISO. Right: daisy-chain — data ripples through slaves in series; only one CS needed but all slaves must support the SDO passthrough mode.
Independent CS is almost always the right choice. Each slave gets its own CS GPIO; SCK, MOSI, and MISO are shared. To address Slave 1, pull CS1 LOW while keeping CS0 and CS2 HIGH. The downside is one GPIO per device — for many peripherals this is fine, and GPIO expanders (shift registers) can extend it further.
Daisy-chain threads data through the slaves in series: the master's MOSI connects to Slave A's SDI, Slave A's SDO connects to Slave B's SDI, and so on; the final slave's SDO returns to the master's MISO. All slaves share the same SCK and CS. The master sends N×8 bits to fill N shift registers simultaneously. Only works if all devices support this mode — LED driver chains (MAX7219) and some SPI-DAC arrays do; most sensors do not.
Clock frequency limits
SPI can run very fast — 10 MHz to 80 MHz is common on short PCB traces, and some flash devices reach 133 MHz. The practical upper limit is set by:
  • Slave device maximum SCK rate — the datasheet specifies fSCK(max). Exceeding it means the slave's internal flip-flops cannot settle before the sample edge. This is a hard limit.
  • Trace propagation delay and capacitance — each picofarad of trace capacitance and stub capacitance adds RC delay. At 20 MHz (50 ns period), a propagation delay of >10 ns means the data changes partway through the high half of the clock — you may sample transitioning data. Keep SPI traces short (under ~5 cm for >10 MHz), avoid vias, and match MOSI/MISO/SCK trace lengths if possible.
  • Master peripheral clock — SPI clock is derived from the peripheral bus clock via a power-of-two prescaler. You can only achieve rates that are exact divisors: fPCLK/2, /4, /8, /16, etc.
  • Setup and hold times — the slave requires data to be stable for tsu before the sample edge and held for th after it. At high clock rates, trace delays can violate these margins.
Rule of thumb: Keep SCK below fPCLK/4 when routing over more than a few centimetres of trace, or when passing through a connector. The extra margin is cheap and prevents intermittent failures that only appear at temperature extremes.
EFR32 USART in SPI mode
The EFR32 USART peripheral supports synchronous (SPI) mode via USART_InitSync(). The same USART hardware used for UART switches to synchronous mode; SCK and CS routing use the same ROUTELOC/ROUTEPEN mechanism as UART (Series 1) or the GPIO route registers (Series 2). 
#include "em_usart.h"
#include "em_gpio.h"
#include "em_cmu.h"

#define SPI_USART     USART1
#define SPI_CLK       cmuClock_USART1

// Pins (check datasheet for your package and LOC)
#define SCK_PORT      gpioPortC
#define SCK_PIN       8
#define MOSI_PORT     gpioPortC
#define MOSI_PIN      6
#define MISO_PORT     gpioPortC
#define MISO_PIN      7
#define CS_PORT       gpioPortC
#define CS_PIN        9

void spi_init(void) {
    CMU_ClockEnable(SPI_CLK,       true);
    CMU_ClockEnable(cmuClock_GPIO, true);

    // Configure GPIO pins
    GPIO_PinModeSet(SCK_PORT,  SCK_PIN,  gpioModePushPull,  0);
    GPIO_PinModeSet(MOSI_PORT, MOSI_PIN, gpioModePushPull,  0);
    GPIO_PinModeSet(MISO_PORT, MISO_PIN, gpioModeInput,     0);
    GPIO_PinModeSet(CS_PORT,   CS_PIN,   gpioModePushPull,  1); // idle HIGH (deasserted)

    USART_InitSync_TypeDef init = USART_INITSYNC_DEFAULT;
    init.master    = true;
    init.baudrate  = 4000000;   // 4 MHz SPI clock
    init.clockMode = usartClockMode0; // CPOL=0, CPHA=0 (Mode 0)
    init.msbf      = true;      // MSB first
    init.databits  = usartDatabits8;
    init.autoTx    = false;
    USART_InitSync(SPI_USART, &init);

    // Route SCK, MOSI, MISO (Series 1 — adjust LOC for your pinout)
    SPI_USART->ROUTELOC0 = USART_ROUTELOC0_CLKLOC_LOC11
                         | USART_ROUTELOC0_TXLOC_LOC11
                         | USART_ROUTELOC0_RXLOC_LOC11;
    SPI_USART->ROUTEPEN  = USART_ROUTEPEN_CLKPEN
                         | USART_ROUTEPEN_TXPEN
                         | USART_ROUTEPEN_RXPEN;
    // CS is a software GPIO — do not route it through USART
}

// Transfer one byte (full-duplex: sends tx, returns rx)
uint8_t spi_transfer(uint8_t tx) {
    return USART_SpiTransfer(SPI_USART, tx);
}

// Read one register from a device that uses cmd | 0x80 for reads
uint8_t spi_read_reg(uint8_t reg) {
    GPIO_PinOutClear(CS_PORT, CS_PIN);          // assert CS
    spi_transfer(reg | 0x80);                   // send address byte (read flag)
    uint8_t val = spi_transfer(0x00);           // clock in data byte
    GPIO_PinOutSet(CS_PORT, CS_PIN);            // deassert CS
    return val;
}

// Write one register
void spi_write_reg(uint8_t reg, uint8_t val) {
    GPIO_PinOutClear(CS_PORT, CS_PIN);          // assert CS
    spi_transfer(reg & 0x7F);                  // send address byte (write: MSB clear)
    spi_transfer(val);                          // send data byte
    GPIO_PinOutSet(CS_PORT, CS_PIN);            // deassert CS
}
USART_SpiTransfer() is a blocking call that writes the byte into the TX buffer, waits for the USART to clock out all 8 bits, and returns the byte that arrived on MISO simultaneously. For high-throughput transfers (e.g. writing a full frame to a display), use DMA: set up an LDMA descriptor with ldmaPeripheralSignal_USART1_TXBL as the trigger and let the DMA feed the TX buffer without CPU involvement.
SPI mode for Series 2. On EFR32 Series 2 devices, the peripheral is EUSART operating in synchronous mode. The init structure becomes EUSART_SpiInit_TypeDef (or use EUSART_SpiAdvancedInit_TypeDef for auto-CS). GPIO routing uses the same GPIO->USARTROUTE[n] registers as Series 2 UART, with the addition of a CLKROUTE field.
