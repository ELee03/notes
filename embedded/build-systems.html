Build Systems & Toolchains

GCC arm-none-eabi · Makefiles · Linker Scripts · Startup Code · CMake · CMSIS · Firmware CI

Before any of the hardware-interfacing knowledge in this cluster becomes useful, you need a build system: the toolchain that compiles C to machine code, the linker script that maps that code into the right memory regions, and the startup file that initializes the processor state before main() is called. IDEs (Simplicity Studio, STM32CubeIDE, Keil µVision) hide this machinery, but understanding it is essential for debugging strange linker errors, optimizing binary size, understanding the generated assembly, and setting up automated firmware CI pipelines.

The GCC ARM Toolchain

The canonical open-source toolchain for ARM Cortex-M is arm-none-eabi-gcc — part of the GNU Arm Embedded Toolchain distributed by ARM Ltd. ("none" = bare-metal, no OS; "eabi" = Embedded Application Binary Interface, the ABI for ARM embedded systems).

The toolchain includes:

  • arm-none-eabi-gcc — C/C++ compiler (compiles + links)
  • arm-none-eabi-as — assembler
  • arm-none-eabi-ld — linker (usually invoked through gcc)
  • arm-none-eabi-objcopy — converts ELF → binary/hex/srec
  • arm-none-eabi-objdump — disassembler, section info
  • arm-none-eabi-nm — symbol table viewer
  • arm-none-eabi-size — section size report (text/data/bss)
  • arm-none-eabi-gdb — debugger front-end (connects to OpenOCD)

Key compiler flags for Cortex-M:

# Cortex-M4 with FPU (EFR32MG series, STM32F4)
CFLAGS  = -mcpu=cortex-m4 -mthumb -mfpu=fpv4-sp-d16 -mfloat-abi=hard
# Cortex-M33 (EFR32xG21)
CFLAGS  = -mcpu=cortex-m33 -mthumb -mfpu=fpv5-sp-d16 -mfloat-abi=hard

# General embedded flags
CFLAGS += -Os                   # Optimize for size
CFLAGS += -ffunction-sections   # Place each function in own section → linker can GC unused
CFLAGS += -fdata-sections       # Same for data
CFLAGS += -Wall -Wextra         # Warnings
CFLAGS += -g3                   # Full DWARF debug info (stripped in final binary)
CFLAGS += -DNDEBUG              # Disable assert() in release

# Linker flags
LDFLAGS = -Wl,--gc-sections     # Remove unused sections (works with -ffunction/data-sections)
LDFLAGS += -Wl,-Map=output.map  # Generate linker map (invaluable for debugging)
LDFLAGS += -Wl,--cref           # Cross-reference table in map
LDFLAGS += -nostdlib            # Don't link standard C startup/library (we provide our own)
LDFLAGS += --specs=nano.specs   # Use newlib-nano (smaller C library)

The -mthumb flag generates Thumb/Thumb-2 instructions. All Cortex-M processors use Thumb-2, never the full 32-bit ARM instruction set. -mfloat-abi=hard uses the hardware FPU with floats passed in FPU registers — faster than soft (all software) or softfp (software ABI, hardware computation).

Compilation pipeline:

main.c  →  [gcc -c]  →  main.o  (ELF relocatable object)
          [nm main.o]           (see symbols: T=text, D=data, B=bss, U=undefined)

main.o + hal_gpio.o + startup.o  →  [ld -T link.ld]  →  firmware.elf
                                                         (ELF with debug info)
firmware.elf  →  [objcopy -O ihex]  →  firmware.hex   (program to flash)
firmware.elf  →  [objcopy -O binary] → firmware.bin   (raw binary)
Makefile Anatomy

A Makefile defines rules: target: prerequisites → recipe. Make rebuilds targets whose prerequisites are newer. For firmware, a minimal but complete Makefile structure:

# Toolchain
CC      = arm-none-eabi-gcc
OBJCOPY = arm-none-eabi-objcopy
SIZE    = arm-none-eabi-size

# Project structure
TARGET  = firmware
BUILD   = build
SRC_DIR = src
INC_DIR = include

# Sources: list all .c files (or use wildcard with care)
SOURCES = $(SRC_DIR)/main.c \
          $(SRC_DIR)/hal_gpio.c \
          $(SRC_DIR)/startup_efr32mg12.s

# Include paths
INCLUDES = -I$(INC_DIR) -I$(SDK)/CMSIS/Include -I$(SDK)/Device/SiliconLabs/EFR32MG12P/Include

# CPU flags
CPU     = -mcpu=cortex-m4 -mthumb -mfpu=fpv4-sp-d16 -mfloat-abi=hard

CFLAGS  = $(CPU) -Os -ffunction-sections -fdata-sections -Wall $(INCLUDES)
LDFLAGS = $(CPU) -Wl,--gc-sections -Wl,-Map=$(BUILD)/$(TARGET).map \
          -T linker/efr32mg12.ld -nostdlib

# Object files (src/main.c → build/main.o)
OBJECTS = $(patsubst %.c,$(BUILD)/%.o, \
           $(patsubst %.s,$(BUILD)/%.o, \
           $(SOURCES)))

# --- Top-level targets ---

all: $(BUILD)/$(TARGET).hex

$(BUILD)/$(TARGET).hex: $(BUILD)/$(TARGET).elf
	$(OBJCOPY) -O ihex $< $@
	$(SIZE) $<

$(BUILD)/$(TARGET).elf: $(OBJECTS)
	$(CC) $(LDFLAGS) -o $@ $^

# Compile C sources
$(BUILD)/%.o: %.c | $(BUILD)
	$(CC) $(CFLAGS) -c -o $@ $<
	# Also generate dependency file so headers are tracked:
	$(CC) $(CFLAGS) -MM -MT $@ $< > $(BUILD)/$*.d

# Assemble startup
$(BUILD)/%.o: %.s | $(BUILD)
	$(CC) $(CPU) -c -o $@ $<

# Create build directory
$(BUILD):
	mkdir -p $(BUILD)/$(SRC_DIR)

# Include dependency files (if they exist)
-include $(OBJECTS:.o=.d)

clean:
	rm -rf $(BUILD)

flash: $(BUILD)/$(TARGET).hex
	openocd -f interface/jlink.cfg -f target/efr32.cfg \
	  -c "program $< verify reset exit"

.PHONY: all clean flash

Key patterns:

  • Automatic variables: $@ = target, $< = first prerequisite, $^ = all prerequisites.
  • Dependency tracking: -MM generates .d files listing header dependencies. Without this, changing a header does not trigger recompilation. The -include at the bottom silently skips missing .d files (first build).
  • Order-only prerequisite | $(BUILD): create the build directory before compiling, but don't remake objects just because the directory timestamp changed.
  • Linker map file: always generate one (-Wl,-Map=...). It shows exactly where every symbol lands in memory — essential for debugging flash overflow, unexpected .bss growth, or finding which library is pulling in an unexpected function.
Linker Scripts

The linker script (.ld file) tells the linker where to place sections in the target's memory map. Without one, the linker has no idea that flash starts at 0x00000000 and SRAM at 0x20000000 on your specific device.

/* EFR32MG12P432F1024GL125 — 1 MB flash, 256 KB RAM */

MEMORY
{
  FLASH (rx)  : ORIGIN = 0x00000000, LENGTH = 1024K
  SRAM  (rwx) : ORIGIN = 0x20000000, LENGTH = 256K
}

ENTRY(Reset_Handler)   /* Entry point — used by debugger, not hardware */

SECTIONS
{
  /* ---- Flash ---- */

  .isr_vector :         /* Vector table MUST be at flash base (0x0) */
  {
    KEEP(*(.isr_vector))
  } > FLASH

  .text :               /* Code and read-only data */
  {
    *(.text*)
    *(.rodata*)
    . = ALIGN(4);
    _etext = .;         /* Exported symbol: end of text (used by startup to copy .data) */
  } > FLASH

  /* ---- Initialized data: lives in flash, copied to SRAM at startup ---- */
  .data :
  {
    _sdata = .;         /* Start of .data in SRAM (copy destination) */
    *(.data*)
    . = ALIGN(4);
    _edata = .;         /* End of .data in SRAM */
  } > SRAM AT > FLASH   /* AT > FLASH = load address is in flash */
  _sidata = LOADADDR(.data);  /* Source address in flash (copy source) */

  /* ---- BSS: zero-initialized — NOT stored in flash ---- */
  .bss :
  {
    _sbss = .;
    *(.bss*)
    *(COMMON)
    . = ALIGN(4);
    _ebss = .;
  } > SRAM

  /* ---- Stack at top of SRAM ---- */
  ._stack :
  {
    . = ALIGN(8);
    _stack_end = .;      /* Low address of stack (stack grows down) */
    . = . + 0x2000;      /* 8 KB stack */
    . = ALIGN(8);
    _stack_top = .;      /* Initial SP value — placed in vector table slot 0 */
  } > SRAM

  /* ---- Heap grows up from after BSS ---- */
  ._heap :
  {
    _heap_start = .;
    . = . + 0x4000;      /* 16 KB heap (newlib malloc arena) */
    _heap_end = .;
  } > SRAM
}

The AT > FLASH syntax is critical: it tells the linker to store the initial values of .data variables in flash (load address), but reserve SRAM space for them at runtime (virtual address). The startup code copies them from _sidata (flash) to _sdata–_edata (SRAM) before calling main().

The linker map file (generated by -Wl,-Map) shows the final layout — each section's VMA (runtime address), LMA (load address in flash), and size. It also shows which object files contribute to each section, and a cross-reference of all symbols. When you hit "binary too large for flash" or "not enough RAM," the map file tells you exactly who is responsible.

The Startup File

The startup file (startup_device.s or startup_device.c) does the work between processor reset and main(). It is usually provided by the silicon vendor but understanding it is essential.

/* startup_efr32.s — minimal ARM Cortex-M startup in GNU assembler */

  .syntax unified
  .thumb

/* === Vector table === */
  .section .isr_vector, "a", %progbits
  .word _stack_top         /* 0x00: Initial Stack Pointer */
  .word Reset_Handler      /* 0x04: Reset */
  .word NMI_Handler        /* 0x08: NMI */
  .word HardFault_Handler  /* 0x0C: HardFault */
  .word MemManage_Handler  /* 0x10: MemManage */
  .word BusFault_Handler   /* 0x14: BusFault */
  .word UsageFault_Handler /* 0x18: UsageFault */
  /* ... 240 IRQ slots follow for external interrupts ... */

/* === Reset handler: runs on every reset/power-on === */
  .section .text.Reset_Handler
  .weak Reset_Handler
  .type Reset_Handler, %function
Reset_Handler:
  /* 1. Set stack pointer (already done by hardware reading vector[0], but make explicit) */
  ldr   r0, =_stack_top
  msr   msp, r0

  /* 2. Copy .data from flash (LMA) to SRAM (VMA) */
  ldr   r1, =_sidata    /* source: init values in flash */
  ldr   r2, =_sdata     /* dest start */
  ldr   r3, =_edata     /* dest end */
copy_data:
  cmp   r2, r3
  bge   zero_bss
  ldr   r0, [r1], #4
  str   r0, [r2], #4
  b     copy_data

  /* 3. Zero .bss */
zero_bss:
  ldr   r2, =_sbss
  ldr   r3, =_ebss
  mov   r0, #0
zero_loop:
  cmp   r2, r3
  bge   call_init
  str   r0, [r2], #4
  b     zero_loop

  /* 4. Call C++ constructors (__libc_init_array handles init_array section) */
call_init:
  bl    __libc_init_array

  /* 5. Jump to main */
  bl    main

  /* 6. If main() returns (shouldn't), loop forever */
loop_forever:
  b     loop_forever

What the hardware does automatically on reset: loads the initial stack pointer from address 0x00000000 (vector[0]), then branches to the address in 0x00000004 (vector[1] = Reset_Handler). Everything after that is software.

Common startup extensions (provided by CMSIS/SDK):

  • System clock initialization: SystemInit() called before main() — configures PLL, wait states, power supplies to the CPU's operating frequency. If you call hardware too early (before SystemInit()), peripherals may run at the wrong clock.
  • FPU enable: Cortex-M4/M7 FPU must be enabled in firmware — it is disabled by default after reset (CPACR register). CMSIS SystemInit() usually handles this.
  • Memory protection unit (MPU): If used, configured before main().
CMake for Embedded

CMake is now the de-facto standard build system for larger embedded projects and is natively supported by VS Code + CMake Tools extension, CLion, and more. The key addition for embedded is a toolchain file.

# toolchain-arm-none-eabi.cmake
set(CMAKE_SYSTEM_NAME       Generic)
set(CMAKE_SYSTEM_PROCESSOR  arm)

set(CMAKE_C_COMPILER    arm-none-eabi-gcc)
set(CMAKE_CXX_COMPILER  arm-none-eabi-g++)
set(CMAKE_ASM_COMPILER  arm-none-eabi-gcc)
set(CMAKE_OBJCOPY       arm-none-eabi-objcopy)
set(CMAKE_SIZE          arm-none-eabi-size)

set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_PACKAGE ONLY)
# CMakeLists.txt
cmake_minimum_required(VERSION 3.20)
project(MyFirmware C ASM)

set(CPU_FLAGS "-mcpu=cortex-m4 -mthumb -mfpu=fpv4-sp-d16 -mfloat-abi=hard")

add_executable(firmware
  src/main.c
  src/hal_gpio.c
  startup/startup_efr32mg12.s)

target_compile_options(firmware PRIVATE
  ${CPU_FLAGS} -Os -ffunction-sections -fdata-sections -Wall)

target_link_options(firmware PRIVATE
  ${CPU_FLAGS}
  -Wl,--gc-sections
  -Wl,-Map=${CMAKE_BINARY_DIR}/firmware.map
  -T ${CMAKE_SOURCE_DIR}/linker/efr32mg12.ld
  -nostdlib --specs=nano.specs)

target_include_directories(firmware PRIVATE
  include
  ${SDK}/CMSIS/Include
  ${SDK}/Device/SiliconLabs/EFR32MG12P/Include)

# Post-build: generate .hex and print size
add_custom_command(TARGET firmware POST_BUILD
  COMMAND ${CMAKE_OBJCOPY} -O ihex $<TARGET_FILE:firmware> firmware.hex
  COMMAND ${CMAKE_SIZE} $<TARGET_FILE:firmware>
  COMMENT "Generating firmware.hex")

Build:

cmake -B build -DCMAKE_TOOLCHAIN_FILE=toolchain-arm-none-eabi.cmake
cmake --build build -j4

CMake advantages over raw Makefiles: automatic dependency tracking, out-of-source builds, easy library management, IDE integration. Disadvantages: more complex, another layer of abstraction. For small projects (<10 source files), a Makefile is often clearer.

CMSIS and Vendor SDKs

CMSIS (Cortex Microcontroller Software Interface Standard) is ARM's portable abstraction layer — a set of header files and utility functions that provide a common interface to Cortex-M core features regardless of which vendor's chip you use.

CMSIS components relevant to firmware development:

  • CMSIS-Core: core_cm4.h, core_cm33.h, etc. — portable access to NVIC, SysTick, MPU, SCB, FPU, and system registers. Defines __NVIC_SetPriority(), __get_MSP(), __DSB(), __ISB(), etc.
  • CMSIS-DSP: Optimized DSP functions using Cortex-M SIMD instructions — FFT, FIR/IIR filters, matrix operations, statistics. Significantly faster than naive C implementations.
  • CMSIS-RTOS2: Portable RTOS API (thread, mutex, semaphore, message queue) — allows switching between FreeRTOS and other RTOS implementations without code changes.

Vendor SDKs (Silicon Labs GSDK, STM32 HAL, NXP SDK) build on top of CMSIS. They add:

  • Device-specific header files (efr32mg12p432f1024gl125.h) — memory-mapped register definitions as C structs and bitfield unions
  • HAL/LL driver libraries for each peripheral
  • Startup files, linker scripts, system initialization code
  • Middleware (USB stacks, file systems, BLE protocol stacks)
  • Board support packages (BSPs) for evaluation boards

Register access pattern (from the device header):

/* CMSIS-style: access GPIO port A output register */
GPIO->P[0].DOUT |= (1 << 5);   /* Set PA5 high */

/* Equivalent raw pointer (what the header abstracts): */
*(volatile uint32_t *)(0x4000A010) |= (1 << 5);  /* GPIOA DOUT register */
Binary Utilities

These tools are used constantly when diagnosing build problems:

# Memory layout: text/data/bss sizes
arm-none-eabi-size firmware.elf
# Output:
#    text     data      bss      dec      hex  filename
#   45312      444     2048    47804     babb  firmware.elf
# text = flash, data = flash (init values) + RAM, bss = RAM only

# Disassemble a function
arm-none-eabi-objdump -d -S firmware.elf | grep -A 30 "<main>"
# -S interleaves C source lines (requires -g debug info)

# List all symbols with addresses
arm-none-eabi-nm --size-sort firmware.elf | tail -20
# Shows biggest functions/variables — great for finding flash hogs

# Show all sections with addresses and sizes
arm-none-eabi-objdump -h firmware.elf

# Convert ELF → raw binary (for bootloader sector writes)
arm-none-eabi-objcopy -O binary firmware.elf firmware.bin

# Convert ELF → Intel HEX (for OpenOCD/J-Link programming)
arm-none-eabi-objcopy -O ihex firmware.elf firmware.hex

# Strip debug info from ELF (reduce file size for release)
arm-none-eabi-objcopy --strip-debug firmware.elf firmware_release.elf

The linker map file (.map) is the most useful single file for understanding your build. Read it with a text editor. Key sections:

  • Memory configuration: shows your MEMORY regions with their sizes and current usage
  • Linker script and input file list: every .o and .a that was linked
  • Section mapping: which objects contributed to each output section, at which addresses
  • Cross-reference table (with --cref): for each symbol, which files define and reference it — invaluable for finding multiple definitions
CI for Firmware

Firmware benefits from the same continuous integration practices as software: every commit triggers a build, and build failure blocks merge. For embedded, CI typically means:

  • Build verification: compiler errors and linker errors caught on every push, regardless of target hardware. Use a Docker container with the arm-none-eabi toolchain installed.
  • Static analysis: cppcheck or clang-tidy to catch common C errors (null pointer dereference, buffer overflow, signed/unsigned comparison).
  • Size regression tracking: fail the build if binary size exceeds a threshold (prevents gradual flash creep). Parse arm-none-eabi-size output and compare against a stored baseline.
  • Unit tests on host: pure logic functions (protocol parsers, state machines, CRC calculation) can be tested on x86 with a normal GCC and a test framework like Unity or CMock. Abstracts hardware via HAL mocks. Fast feedback without hardware.
  • Hardware-in-the-loop (HIL): for critical path testing, a CI runner connected to target hardware flashes the firmware and runs a test script. More infrastructure cost; used in production firmware teams.

Minimal GitHub Actions workflow:

# .github/workflows/build.yml
name: Firmware Build
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    container: ubuntu:22.04
    steps:
      - uses: actions/checkout@v4
      - name: Install toolchain
        run: |
          apt-get update -q
          apt-get install -y gcc-arm-none-eabi binutils-arm-none-eabi make
      - name: Build
        run: make -C firmware all
      - name: Check size
        run: |
          arm-none-eabi-size firmware/build/firmware.elf
          # Fail if flash > 900 KB (out of 1 MB, leave 100 KB for bootloader)
          python3 scripts/check_size.py firmware/build/firmware.elf --flash-limit 921600
Worked Examples
Example 1 — Diagnosing a "undefined reference to `__errno'" linker error
Adding #include <stdio.h> and calling printf() to an embedded project produces: undefined reference to `_write' and undefined reference to `_sbrk'. Explain what these symbols are and how to resolve them.

newlib (the C library used by arm-none-eabi) provides printf() but delegates low-level I/O to stub functions you must implement. These are the syscall stubs:

  • _write(int fd, char* buf, int len): output characters — for printf to produce output, this must be implemented. Typically redirects to UART or ITM.
  • _sbrk(ptrdiff_t incr): heap allocation — malloc() calls this to extend the heap. Must return a pointer to new heap memory, bounded by the heap end symbol from the linker script.

Implement them as:

#include <sys/stat.h>
#include "hal_uart.h"

/* Redirect stdout to UART */
int _write(int fd, char *buf, int len) {
  for (int i = 0; i < len; i++) HAL_UART_Transmit(buf[i]);
  return len;
}

/* Minimal sbrk for newlib heap */
extern char _heap_start[];   /* from linker script */
extern char _heap_end[];
static char *heap_ptr = _heap_start;

void *_sbrk(ptrdiff_t incr) {
  char *prev = heap_ptr;
  if (heap_ptr + incr > _heap_end) return (void *)-1;  /* OOM */
  heap_ptr += incr;
  return prev;
}

Alternative: link with --specs=nano.specs --specs=nosys.specs. The nosys.specs provides stub implementations that do nothing (writes are discarded, sbrk returns -1). This silences the errors but means printf produces no output and malloc always fails.

Example 2 — Reading the linker map to debug flash overflow
Adding a new feature causes "region FLASH overflowed." The linker map shows .rodata grew from 2 KB to 18 KB. Where should you look?

.rodata (read-only data, stored in flash) holds: string literals, constant arrays, lookup tables, and virtual function tables (C++). A jump from 2 KB to 18 KB suggests one of:

  1. A large lookup table or bitmap: a const uint8_t font_data[8192] or similar. Find it in the map's section listing — the contributing object file will be named.
  2. String literals accumulated: every printf("some message"), error string, or log tag adds to .rodata. Use arm-none-eabi-nm --size-sort firmware.elf | grep " r " to list read-only symbols by size.
  3. Floating-point library pulled in: linking anything that uses printf("%f", ...) or sprintf with floats pulls in the newlib float formatter — 6–10 KB. Use arm-none-eabi-nm firmware.elf | grep printf to check if multiple printf variants are linked.

Mitigations: move constant arrays to external flash if available; replace printf with a stripped-down tiny_printf; use -fno-exceptions and -fno-rtti in C++ to avoid vtable bloat; ensure -ffunction-sections -fdata-sections -Wl,--gc-sections are in use so unused constants are removed.

Example 3 — Why does a variable initialized to 0 work, but one initialized to 42 resets to 0 after a warm reset?
A global variable uint32_t counter = 42 behaves correctly after cold power-on, but after a software reset (NVIC_SystemReset()) it reads as 0. Explain.

After power-on, the startup code copies .data initial values from flash (_sidata) to SRAM (_sdata–_edata). So counter gets the value 42 from flash. This is correct.

After a software reset (NVIC_SystemReset()), the processor jumps to Reset_Handler again. But SRAM is not cleared by a software reset — only a power-cycle clears SRAM. The startup code then:

  1. Copies .data initial values from flash → SRAM. counter is set to 42 again. ✓
  2. Zeroes .bss — but counter is in .data, not .bss, so this doesn't affect it.

If the variable is reading 0 after warm reset despite correct startup code, the likely culprit is the startup code is not running on warm reset. This happens when:

  • A debugger is used that doesn't reset the PC to Reset_Handler on reset (skips startup code)
  • The VTOR register is not pointing to the correct vector table after a bootloader hands off
  • A custom reset sequence jumps directly to main() bypassing startup

If startup code is running but writes the wrong value: check that _sidata in the linker script is correctly defined as LOADADDR(.data) (the flash address of the init values) and that the copy loop uses the correct length (_edata - _sdata bytes, word-aligned).

Food for Thought
  • The linker flag -Wl,--gc-sections removes unreferenced sections from the binary. It requires -ffunction-sections and -fdata-sections to be useful. Why can't the linker simply remove unused code and data without those flags?
    Without -ffunction-sections, the compiler places multiple functions into the same .text section of an object file. The linker's granularity for dead-code elimination is the section — it can discard an entire section, but not individual functions within a section. If foo() and bar() both live in .text, and foo() is referenced but bar() is not, the linker must keep the entire section (including bar()). With -ffunction-sections, each function gets its own section (.text.foo, .text.bar), so the linker can eliminate .text.bar independently. The same logic applies to data with -fdata-sections. On a typical embedded project this can reduce binary size by 10–40%.
  • CMake generates Makefiles (on Unix) or Ninja build files. If the output of CMake is a Makefile, what is CMake actually adding — why not write the Makefile directly?
    CMake handles things that are painful to do in raw Makefiles: automatic dependency tracking (no need to manually write .d file generation), cross-platform builds (the same CMakeLists.txt generates Makefiles on Linux, .vcxproj on Windows, Xcode projects on macOS), IDE integration (VS Code, CLion consume CMake natively), hierarchical project structure with add_subdirectory(), package management (find_package() for locating libraries), and the ability to generate compile_commands.json (consumed by clangd for in-editor autocompletion and navigation). For a simple single-directory project, the raw Makefile is clearer and faster. For anything with multiple libraries, conditional compilation, or multi-developer cross-platform support, CMake pays for its complexity.
  • The startup file zeroes the .bss section on every reset. If SRAM is already zero after a power-cycle (as it is on most MCUs), why bother zeroing it in startup?
    SRAM retains its contents across software resets and resets from debugging. If your startup code doesn't zero .bss, a variable declared as static int count; (which the C standard guarantees is initialized to 0) may contain garbage from a previous run after a soft reset. The C standard mandates zero-initialization of static-duration variables with no explicit initializer — the startup code is what implements this guarantee on bare metal. Without it, your code has undefined behaviour in any scenario involving a warm reset, a watchdog reset, or repeated debugging sessions. A power-cycle may happen to zero SRAM on your specific chip, but relying on this is non-portable and masks bugs.
Common Pitfalls
  • Forgetting -ffunction-sections -fdata-sections -Wl,--gc-sections. These are not defaults — without them, every function and data object from every library you link ends up in the binary even if never called. Can double or triple binary size versus an optimized build.
  • Using float with printf inadvertently. Linking printf("%f", x) pulls in the full newlib floating-point formatter (~6–10 KB). Use printf("%d", (int)(x*100)) for fixed-point display, or link with -u _printf_float only when you genuinely need it and have the flash budget.
  • Putting interrupt handlers in a different file from their declaration without KEEP(). If --gc-sections is on and the ISR is not referenced from code (it's only called by hardware via the vector table), the linker may garbage-collect it. Ensure the vector table references the ISR by name, or mark the ISR section with KEEP() in the linker script.
  • Stack and heap collision. If the stack grows down into the heap, you get corruption — classic symptom is malloc returning bad pointers after heavy function call depth. Add a stack canary (fill the stack region with a known value at startup, check it periodically) and monitor _stack_end to see how close you are.
  • Ignoring compiler warnings. Embedded C has no exceptions. A signed/unsigned comparison warning or implicit function declaration warning will cause real bugs in production. Build with -Wall -Wextra -Werror and fix every warning.
