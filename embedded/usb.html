12 · USB Fundamentals
Enumeration · Descriptors · Transfer Types · CDC-ACM · USB-FS vs USB-HS
USB — Universal Serial Bus — is everywhere: every laptop, microcontroller development board, and medical device with a Type-C connector is running it. But USB is radically more complex than UART, SPI, or I2C. Where those protocols are documented in a few pages, the USB 2.0 specification is 650 pages. Most embedded engineers interact with USB through a protocol stack (a USB device library), not by bit-banging registers. Understanding the fundamentals still matters: it tells you why enumeration fails, why your device is slow, and why that latency spike happens on every 1 ms boundary.
This lesson covers the conceptual model that underlies all USB 2.0 device firmware: what happens at the electrical level, how enumeration works, how the descriptor hierarchy tells the host what kind of device you are, the four transfer types and when to use each, and CDC-ACM as the practical example most embedded engineers reach for first.
Why USB is more complex than other protocols
Every protocol in this cluster so far (UART, SPI, I2C, CAN) follows the same basic model: both sides agree on a format, one side asserts a signal, the other reads it. USB is fundamentally different in two ways:
  • The host is always in charge. USB has no peer-to-peer communication. One host (a PC, SoC, or USB OTG controller) owns the bus and polls every device on a fixed schedule. Devices never initiate a transfer — they only respond to host requests. This means your MCU firmware cannot "send data when ready" the way it can over UART; it can only pre-load a buffer and wait for the host to ask for it.
  • Devices must identify themselves. A UART receiver just starts decoding bytes. A USB host does not know what a device is until it reads a structured set of descriptors from it. The entire process of reading those descriptors, assigning an address, and loading a driver is called enumeration, and it must complete correctly before any data transfer can occur.
Electrical layer: D+ and D−
USB uses a differential pair: D+ and D−. Like CAN, the receiver responds to the differential voltage VD+ − VD−, rejecting common-mode noise. Unlike CAN, USB is not a multi-drop bus — it is a strict host-to-device point-to-point or star topology via a hub.
Speed Bit rate D+ pull-up Common use
Low Speed (LS) 1.5 Mbps D− pull-up HID devices (mouse, keyboard)
Full Speed (FS) 12 Mbps D+ pull-up to 3.3 V CDC, HID, audio — most MCU USB
High Speed (HS) 480 Mbps Negotiated during reset Mass storage, video, high-throughput data
Speed detection happens automatically. When the device connects, the host detects which line is pulled up (FS: D+ high; LS: D− high) to determine the initial speed. A HS-capable device starts at FS and then negotiates up to HS during the reset sequence using a chirp handshake.
USB-FS signal levels are 3.3 V differential. USB-HS uses current-mode signalling (17.78 mA into 45 Ω per line), which is why HS PHYs require an external or integrated transceiver and cannot be done with simple GPIO-level push-pull driving. Most MCUs that claim "USB-HS" integrate the HS PHY on-chip; some require an external ULPI PHY chip.
Enumeration
Enumeration is the conversation that happens between the host and device immediately after connection. It proceeds in a fixed sequence, and if any step fails, the device is never usable — it sits unrecognised in Device Manager or dmesg with an error code.
USB enumeration sequence. Every step is a host-initiated control transfer on endpoint 0. If any step times out or returns an error, enumeration fails and the device is unusable.
The address assigned in SET_ADDRESS is how the host identifies the device for the rest of the session — it is chosen by the host, not the device. After SET_ADDRESS the device firmware must switch its response address before the next transaction, which is why USB device stacks handle this in the interrupt handler with tight timing.
Descriptor hierarchy
A USB device describes itself through a structured hierarchy of descriptors — binary data structures with fixed field layouts. The host reads these to understand what the device is, how to configure it, and how to communicate with it.
One device per bus address. Each device has one or more configurations. Each configuration has one or more interfaces (a logical function). Each interface has endpoint descriptors describing its data pipes.
A few key fields in the device descriptor matter most in firmware:
  • idVendor / idProduct (VID/PID) — 16-bit numbers that identify the manufacturer and product. The host OS uses these to find the right driver. You need a registered VID for commercial products; for prototyping, use a VID/PID pair licensed for open-source use (e.g. from the OpenMoko pool, or buy a sub-licence from a vendor).
  • bDeviceClass / bInterfaceClass — class codes tell the host what kind of device this is. Class 0x02 = CDC (Communications Device Class), 0x03 = HID, 0x08 = Mass Storage. Standard class codes trigger standard OS drivers, meaning no custom driver is needed on the host.
  • bMaxPacketSize0 — the maximum payload size for endpoint 0 (the control endpoint). The host reads this in the first 8-byte GET_DESCRIPTOR and adjusts all subsequent control transfers accordingly.
The four transfer types
USB defines four endpoint transfer types, each with different guarantees on latency, bandwidth, and reliability. Every endpoint has exactly one type, declared in its descriptor.
Type Guaranteed latency? Error recovery Bandwidth Typical use
Control No Retries Up to 20% of frame Enumeration, device configuration (EP0 only)
Bulk No Retries (reliable) Leftover bandwidth Mass storage, CDC data, printer — large reliable transfers
Interrupt Yes (polling interval) Retries Guaranteed slice per frame HID (keyboard, mouse), CDC notification endpoint
Isochronous Yes (every frame) None (drops) Up to 1023 B/frame (FS) Audio, video — timing-critical streams that tolerate drops
The 1 ms USB frame is the heartbeat of the bus. At Full Speed, the host generates a Start-of-Frame (SOF) token every 1 ms. Control and interrupt transfers are scheduled within each frame; bulk transfers fill whatever bandwidth remains. Isochronous endpoints get a guaranteed slot in every frame, but dropped packets are never retransmitted — audio glitches rather than stalls.
Why bulk is not "fast". Bulk has no guaranteed latency. If a high-priority interrupt endpoint fills most of a frame's bandwidth, your bulk transfer waits. In practice, on a lightly loaded USB bus bulk is fast (multiple megabytes per second at FS). On a busy bus with lots of HID and audio traffic, bulk can stall for many milliseconds. If you need deterministic latency, use interrupt transfers.
CDC-ACM: virtual serial port
The most common USB class in embedded development is CDC-ACM — Communications Device Class, Abstract Control Model. It makes your MCU appear to the host OS as a standard serial port (COM port on Windows, /dev/ttyACMx on Linux, /dev/cu.usbmodem... on macOS). No custom driver needed; the OS ships one. This is why development boards can be programmed and debugged over the same USB cable used for power.
CDC-ACM uses two USB interfaces and three endpoints:
  • Interface 0 — CDC Control: One interrupt IN endpoint (the "notification endpoint") used to send line state changes (DTR, DCD) to the host. In practice many implementations never use this and the host still works.
  • Interface 1 — CDC Data: One bulk IN endpoint (device→host data) and one bulk OUT endpoint (host→device data). These carry the actual byte stream.
The host can also send control requests to set the baud rate, data bits, parity, and stop bits via the SET_LINE_CODING request. Your MCU firmware receives these and can apply them to a hardware UART — useful for USB-UART bridge firmware. Or you can ignore them entirely if you are not bridging to a real UART and just use the bulk endpoints as a generic byte pipe. 
// CDC-ACM endpoint configuration (simplified — actual stack handles this)
//
// EP0   Control  (IN + OUT)  — enumeration, SET_LINE_CODING, etc.
// EP1   Interrupt IN         — CDC notification (line state, often unused)
// EP2   Bulk IN              — device → host data bytes
// EP3   Bulk OUT             — host → device data bytes
//
// From firmware's perspective, you just fill EP2's TX buffer when you
// have data to send, and read from EP3's RX buffer when the host writes.
// The USB stack fires a callback when EP3 has new data.
//
// Example: Silicon Labs USB library (em_usb.h)
//
// void usbDataReceived(USB_Status_TypeDef status,
//                      uint32_t xferred,
//                      uint32_t remaining) {
//     if (status == USB_STATUS_OK) {
//         // xferred bytes available in rx_buf
//         process_bytes(rx_buf, xferred);
//         // re-arm the OUT endpoint for next transfer
//         USBD_Read(CDC_EP_DATA_OUT, rx_buf, sizeof(rx_buf), usbDataReceived);
//     }
// }
//
// void usbSend(const uint8_t *data, uint32_t len) {
//     USBD_Write(CDC_EP_DATA_IN, (void*)data, len, NULL);
// }
The key thing to understand is that CDC-ACM bulk transfers are still subject to the USB 1 ms frame boundary. If you call the write function between two SOF tokens, your data goes out in the very next frame — 0 to 1 ms latency. This means CDC-ACM is not suitable as a real-time data path for anything that needs sub-millisecond latency. For that, use an interrupt endpoint (bounded to your declared polling interval, typically 1–16 ms for FS).
VID/PID for CDC-ACM in prototypes. Silicon Labs devices with the official USB PID from their USB library will enumerate as a Silicon Labs virtual COM port — which is fine for development. For a product shipped to customers, you need your own registered VID (from USB-IF) and a product PID. Sharing a VID/PID with another product in the field causes mysterious device conflicts on the host.
