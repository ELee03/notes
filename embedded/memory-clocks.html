Memory & Clock Systems
Embedded Systems · Lesson 2
Flash Memory
Flash is non-volatile — it survives a power cycle. It stores your compiled program (the .text section), read-only data (.rodata), and the initial values of any global variables (.data load region). On power-up, all of this is still there, exactly as the programmer left it.
Flash works by storing charge in floating-gate transistors. Writing to flash is destructive and asymmetric: you can only flip bits from 1 to 0 (program), and to flip them back to 1 you must erase an entire page — typically 2–8 KB — all at once. This means you can never partially overwrite flash. Firmware update schemes and non-volatile storage libraries (like NVM3 on EFR32) are designed around this constraint.
Flash endurance matters. Each page has a finite erase/write cycle life — typically 10,000–100,000 cycles for the embedded flash on an MCU. A naively implemented data-logging loop that writes to the same page on every sensor sample can wear out a page within hours. Storage libraries use wear levelling to spread writes across pages.
The CPU fetches instructions from flash over the AHB bus, but flash is slower than the CPU at high clock speeds. When the CPU is running at 38 MHz and flash has a 70 ns read cycle time, the core would stall every instruction fetch waiting for flash to respond — unless hardware inserts wait states (idle bus cycles) to give flash time to present valid data.
The number of required wait states scales with clock speed. A typical rule for EFR32 Series 1 (M4F):
CPU Clock Flash Wait States Effect
≤ 16 MHz 0 WS Flash keeps up; no stalls
16 – 38.4 MHz 1 WS One idle cycle per fetch; ~5–10% throughput loss
The flash controller inserts wait states automatically once you configure the correct value in the MSC_READCTRL register — but you must set this before raising the clock. If you switch to 38 MHz with zero wait states still configured, the CPU will fetch garbage and your firmware will crash in a completely unpredictable way. The correct order is always: set wait states → raise clock, never the reverse.
SRAM Layout
SRAM holds everything that changes at runtime: global variables, the stack, the heap, peripheral DMA buffers. It is fast (same speed as the CPU bus), but volatile — it loses its contents on every power cycle.
The linker is what assigns every symbol in your program to a specific address in memory. It reads a linker script (e.g., efr32bg22.ld) that defines the size and start address of each memory region, then packs the output sections into those regions. The resulting layout in SRAM is fixed at link time, not at runtime.
Stack overflow is the most common hard-to-debug failure on resource-constrained MCUs. The stack starts at the top of SRAM and grows downward. Deep call chains, large local arrays, or recursive functions can push the stack past the heap and into .bss, silently corrupting global variables. The Cortex-M MPU (memory protection unit) can be configured to trap stack overflows by placing a no-access region at the bottom of the stack — but this requires explicit setup and is not automatic.
Avoid heap on bare-metal. malloc works on most MCU C runtimes, but the heap fragmentation problem is worse on a device with 32 KB of SRAM than on a server. Most production bare-metal firmware pre-allocates all buffers statically and never calls malloc. If you need dynamic-size data structures, use a fixed-size memory pool instead.
The Clock Tree
Every digital operation in an MCU is synchronised to a clock. The CPU needs a clock to advance its pipeline. Peripherals need clocks to drive their state machines. Timers need clocks to count. The clock tree is the network of oscillators, multiplexers, PLLs, and prescalers that generates all of these clocks from one or a few physical sources.
The root sources are:
Crystal oscillators (XO) — a quartz crystal mechanically resonates at a precise frequency. The MCU drives it with an inverting amplifier and the crystal provides frequency selectivity. Crystals are accurate (±10–50 ppm) and stable over temperature, but they take milliseconds to start up and consume more current than RC oscillators. You must place the crystal close to the MCU pins with minimal trace capacitance — a poorly designed crystal circuit will not start reliably.
RC oscillators (RCO) — on-chip resistor-capacitor circuits that oscillate at a factory-trimmed frequency. They start up in microseconds, consume very little current, and cost nothing in BOM. The price is accuracy: RC oscillators drift significantly with temperature (±1–3% over the full range) and age. Fine for general timing; not acceptable for clock-sensitive protocols like UART at high baud rates without calibration.
PLLs (Phase-Locked Loops) — take a low-frequency reference (say, 38.4 MHz from a crystal) and multiply it up to a higher frequency by locking a voltage-controlled oscillator to a divided-down version of itself. PLLs let you run the CPU at 80 MHz from a 38.4 MHz reference, or synthesize a precise 48 MHz USB clock from the same reference. They consume more current than raw oscillators and have a lock-acquisition time of ~100 µs.
The prescaler chain lets peripherals run slower than the CPU without needing separate oscillators. An APB peripheral clock at HCLK/4 means the UART baud rate generator divides a known, slower frequency — reducing register setup complexity and power consumption in that peripheral block while the CPU core still runs at full speed.
EFR32 Clock Management Unit (CMU)
The CMU is the EFR32's clock controller — the single register block that configures all clock sources, mux selections, prescalers, and per-peripheral clock gates. In Series 1 it is memory-mapped at 0x400E4000; in Series 2 the CMU is reorganised but the concept is identical.
The key clock sources on EFR32 Series 1:
HFXO (High-Frequency Crystal Oscillator) — driven by an external 38.4 MHz crystal. This is the most accurate high-frequency source and the reference for any frequency-sensitive protocol (Bluetooth, 802.15.4, UART at high baud). It requires a crystal and two load capacitors externally, and takes ~300 µs to stabilise after being enabled.
HFRCO (High-Frequency RC Oscillator) with DPLL — an on-chip RC oscillator calibrated at the factory to a set of target frequencies (19, 26, 38 MHz, etc.). In Series 2 devices, the HFRCO is paired with a DPLL (digital PLL) that can phase-lock it to the LFXO, yielding much better accuracy without needing the HFXO. Good for CPU operation when power matters more than timing precision.
LFXO (Low-Frequency Crystal Oscillator) — 32.768 kHz (the standard RTC crystal frequency). Extremely low current draw (~100 nA), accurate, and keeps running through deep sleep modes to maintain timekeeping.
LFRCO / ULFRCO — on-chip low-frequency RC oscillators. The ULFRCO runs at approximately 1 kHz and is the only clock source that survives down to Energy Mode 4 (EM4). It keeps the watchdog alive even in near-shutdown states.
A typical startup sequence in C using CMSIS/emlib:
/* 1. Set flash wait states BEFORE raising clock (Series 1) */ MSC->READCTRL = MSC_READCTRL_MODE_WS1; /* 1 wait state for HFCLK > 16 MHz */ /* 2. Enable and wait for HFXO */ CMU_OscillatorEnable(cmuOsc_HFXO, true, true); /* enable, wait for ready */ /* 3. Select HFXO as HF clock source */ CMU_ClockSelectSet(cmuClock_HF, cmuSelect_HFXO); /* 4. Enable clocks to specific peripherals as needed */ CMU_ClockEnable(cmuClock_USART0, true); CMU_ClockEnable(cmuClock_GPIO, true);
Per-peripheral clock gating. The CMU can individually gate the clock to each peripheral. A peripheral with its clock gated draws essentially zero dynamic power and any register writes to it are silently ignored. Always enable a peripheral's clock before configuring it. Conversely, disabling the clock to peripherals you are not using is one of the most effective active-mode power savings available.
Energy Modes (EM0–EM4)
The EFR32 defines five energy modes that progressively shut down more of the chip to reduce current consumption. The right mode for any situation depends on what must still be running while the CPU is idle.
Mode CPU HF Clocks LF Clocks SRAM Typical ICC
EM0 — Active Running On On Full 3–10 mA
EM1 — Sleep Halted On On Full 1–3 mA
EM2 — Deep Sleep Halted Off On Retained 1–10 µA
EM3 — Stop Halted Off Off Retained ~0.4 µA
EM4 — Shutoff Halted Off Off Lost ~20 nA
EM2 is the workhorse for battery-powered wireless devices. The CPU is off and all HF clocks are stopped, but the low-frequency clock tree keeps running. The LETIMER (Low-Energy Timer), RTC, and low-energy UART can continue operating — they wake the CPU via interrupt when data arrives or a deadline expires. A Bluetooth device spending 99% of its time in EM2 can run for months on a coin cell.
EM4 is near-shutdown. SRAM is lost — when the device wakes from EM4 it goes through a full reset sequence. Only the backup RTC and a small number of retention registers survive. EM4 makes sense for a device that should drain essentially no power for hours between rare wakeups (an asset tracker that reports once per hour, for example).
Wake latency is the other side of the trade-off. Exiting EM2 takes ~2–5 µs. Exiting EM3 can take ~30 µs. Exiting EM4 requires a full reset and re-initialisation of all peripherals — hundreds of microseconds. If your application wakes frequently (every few milliseconds for a sensor poll), EM2 or EM1 is appropriate; the wake latency of EM4 would negate any power saving.
