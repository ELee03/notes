MCU Architecture
Embedded Systems · Lesson 1
What Is a Microcontroller?
A microprocessor is a CPU — it processes data but relies on external chips for everything else: RAM, flash, clocks, peripherals. A microcontroller (MCU) integrates all of that onto a single die. One chip contains the CPU core, flash memory for your program, SRAM for runtime data, clock oscillators, and a collection of peripherals — timers, ADCs, communication interfaces — all wired together on the same silicon.
The practical consequence is stark: an MCU can run directly from a coin cell with nothing more than bypass capacitors and a crystal. A microprocessor needs gigabytes of DRAM, a voltage regulator, a separate flash controller, and a board that costs more than the processor itself. For embedded systems — especially battery-powered wireless devices — this integration is everything.
The EFR32 family (Silicon Labs) pushes this further: it integrates a 2.4 GHz radio transceiver alongside the CPU core and peripherals, making it a System-on-Chip (SoC). The radio, CPU, RAM, flash, and peripherals are all on one die. No external RF IC needed.
The ARM Cortex-M Family
ARM Ltd. does not make chips — it designs CPU cores and licenses them as intellectual property. Silicon vendors (Silicon Labs, STMicroelectronics, NXP, Nordic, etc.) then integrate those cores into their own SoCs, adding their own memory, radio blocks, and peripherals around it. The core itself is the same across vendors; only the surrounding peripherals differ.
The Cortex-M profile is the embedded family — optimized for deterministic real-time operation, low power, and low cost. Within it, the important members are:
Core Pipeline FPU TrustZone Notes
M0/M0+ 2–3 stage No No Smallest area; Thumb ISA only
M3 3 stage No No Thumb-2; hardware divide; branch prediction
M4 3 stage Optional No DSP extensions; SIMD; used in EFR32xG12/13
M7 6 stage Yes No High-performance; dual-issue; cache
M33 3–5 stage Optional Yes Successor to M3/M4; used in EFR32xG21/22/23/24
M55 5 stage Yes Yes Helium (SIMD/ML); highest-end Cortex-M
The M4 and M33 are the two cores you will encounter most often in the EFR32 family. The M4 adds DSP extensions over the M3 — SIMD multiply-accumulate instructions that let you process sensor data and run FIR filters without a dedicated DSP chip. The M33 is architecturally newer: it shares the M4's instruction set (plus TrustZone security extensions) but with a more efficient pipeline and a cleaner memory protection unit design.
The Register File
The Cortex-M has 16 general-purpose registers (R0–R15), each 32 bits wide. Most are genuinely general-purpose: the compiler can use them freely for temporaries, function arguments, and return values. Three are architecturally special and you need to understand them to read assembly or debug exceptions.
R13 (SP) is banked. There are actually two stack pointers: the Main Stack Pointer (MSP) and the Process Stack Pointer (PSP). In bare-metal firmware you almost always use only the MSP. The PSP becomes relevant when you run an RTOS — each task gets its own stack using the PSP, while the MSP is reserved for interrupt handlers and the OS kernel.
R14 (LR) stores the return address when a function is called via the BL (branch-with-link) instruction. The core automatically writes PC + 4 into LR before branching. On exception entry, the core loads a special EXC_RETURN value into LR instead of a real address — this encodes which stack and mode to return to.
xPSR is a composite of three sub-registers: the Application PSR (APSR, holds N/Z/C/V/Q flags), the Interrupt PSR (IPSR, holds the current exception number), and the Execution PSR (EPSR, holds the Thumb bit and IT/ICI fields). You almost always read/write them as a combined 32-bit word via the MRS and MSR instructions.
Processor Modes and Privilege
The Cortex-M operates in one of two modes and one of two privilege levels, giving four combinations — though one combination is forbidden by hardware.

Thread Mode is where your application code runs. Handler Mode is where exception and interrupt handlers run. The key rule: Handler Mode is always privileged. The hardware enforces this. Thread Mode can be either privileged or unprivileged, controlled by the nPRIV bit in the CONTROL register.
In bare-metal firmware you typically stay in privileged Thread Mode the entire time — there is no OS to protect kernel memory from user code, so the separation is unnecessary. When you run an RTOS, the OS may drop tasks into unprivileged Thread Mode (using the PSP), so a misbehaving task cannot corrupt kernel state or peripheral registers directly.
You cannot promote yourself. Unprivileged Thread Mode code cannot set CONTROL.nPRIV = 0 to regain privilege — the write is silently ignored. The only way back to privileged is via an exception, where the handler (running privileged in Handler Mode) can choose to clear nPRIV before returning to Thread Mode. This is how an RTOS syscall works: a task triggers a SVC (Supervisor Call) exception, the privileged SVC handler runs, does the privileged work, then returns.
The Memory Map
The Cortex-M uses a 32-bit address space: 4 GB, from 0x00000000 to 0xFFFFFFFF. ARM defines a fixed partitioning of this space — every Cortex-M device from every vendor follows the same layout. This is deliberate: it lets compilers, RTOS code, and debuggers be written once and work everywhere.
The peripheral region is the most vendor-specific part. Every Cortex-M places peripherals somewhere in the 0x40000000–0x5FFFFFFF range, but the exact address of each peripheral block (UART0, SPI1, GPIO, etc.) differs per chip. Reading a peripheral register is literally just a load/store to a fixed memory address — there is no separate I/O address space.
The System region (0xE0000000+) is the same on every Cortex-M from every vendor. The NVIC, SysTick timer, System Control Block (SCB), and MPU all live at fixed ARM-defined addresses here. This is why CMSIS (ARM's low-level library) can define NVIC_EnableIRQ() once and have it work on any Cortex-M chip.
Peripheral registers behave like memory but are not memory. Reads and writes may have side effects — reading the UART data register consumes a byte from the hardware FIFO. The compiler must not reorder or cache these accesses. This is why peripheral register pointers are always declared volatile in C.
The Reset Sequence
When power comes on (or a reset is asserted), the core does not start at address zero and run whatever byte happens to be there. It follows a specific boot protocol defined by the architecture.
The vector table lives at address 0x00000000 by default (or wherever the Vector Table Offset Register, VTOR, points). It is an array of 32-bit words. The first two entries are special:
Entry 0 (0x00000000): Initial value of MSP — the core loads this into R13 on reset Entry 1 (0x00000004): Address of the Reset Handler — the core jumps here after loading MSP
After that, the core is executing C-startup code (usually in a file called startup_efr32xxx.s or similar). The startup code must do three things before it can safely call main():

The reason .data must be copied from flash is that flash is non-volatile — SRAM is wiped on every power cycle. Any global variable with an initial value (like int count = 10;) has that initial value baked into the flash image at link time. The startup code copies the entire .data section from flash into the .data region in SRAM before main() runs, so your globals have the right values.
SystemInit() (from CMSIS) runs before main() and configures the clocks — enabling the PLL, setting the CPU core clock frequency, and configuring flash wait states to match the new clock speed. If you skip it or get it wrong, the chip may run at a default low-speed clock or hang entirely when trying to fetch fast instructions from slow flash.
EFR32 Specifics
The EFR32 is Silicon Labs' flagship wireless SoC family. All members share the same general architecture — ARM Cortex-M core, on-chip flash and SRAM, standard APB/AHB peripheral bus, and an integrated sub-GHz or 2.4 GHz radio — but the core and clock frequency vary by series:
Series Examples Core Max Clock Flash / SRAM
xG12 EFR32BG12, MG12, FG12 Cortex-M4F 38.4 MHz up to 1 MB / 256 KB
xG13 EFR32BG13, MG13, FG13 Cortex-M4F 38.4 MHz up to 512 KB / 64 KB
xG21 EFR32BG21, MG21 Cortex-M33 80 MHz up to 1 MB / 96 KB
xG22 EFR32BG22, MG22 Cortex-M33 38.4 MHz up to 512 KB / 32 KB
xG23 EFR32BG23, MG23, FG23 Cortex-M33 78 MHz up to 512 KB / 64 KB
xG24 EFR32BG24, MG24 Cortex-M33 78 MHz up to 1.5 MB / 256 KB
Series 1 devices (xG12, xG13) use the Cortex-M4F — the same core covered in detail by Yiu's reference. Series 2 devices (xG21 and newer) use the Cortex-M33, which introduces TrustZone security extensions and a more efficient pipeline but is architecturally similar enough that everything in this lesson applies to both. The main practical difference is that Series 2 devices require the Secure/Non-secure memory partitioning to be configured (or disabled) in the FLASH controller before your application runs.
Simplicity Studio and CMSIS. Silicon Labs wraps the EFR32 in the CMSIS framework — the same register-access macros and naming conventions ARM specifies for all Cortex-M devices. NVIC_EnableIRQ(USART0_RX_IRQn) works on an EFR32 the same way it does on an STM32. The EFR32-specific header files (e.g., efr32bg22.h) define the peripheral base addresses and register bitfield macros; you almost never hardcode hex addresses directly.
The EFR32 also exposes the ARM Serial Wire Debug (SWD) interface via two pins (SWDIO and SWDCLK). This gives you full read/write access to the entire memory map — including peripheral registers and SRAM — from a debug probe while the chip is running. Lesson 15 covers debugging in detail; for now, just know that this is how you flash and debug the chip.
