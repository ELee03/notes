5 · GPIO
Output Modes · Input Pull-ups · Schmitt Trigger · Alternate Functions · Pin Routing
General Purpose Input/Output — GPIO — is the most fundamental peripheral in any microcontroller. Every other peripheral on the chip (UART, SPI, ADC, timers) ultimately exposes itself to the outside world through GPIO pins. Understanding how GPIO pads work at the circuit level explains why the same pin can function as a logic output, an I2C line, an ADC input, or a PWM channel — and why certain combinations of signals require specific electrical configurations.
A GPIO pad is a small bidirectional interface between the silicon and the package pin. Inside the pad cell there is an output driver, an input receiver, optional pull resistors, a Schmitt trigger, and a multiplexer that selects between the GPIO function and one of several peripheral functions. The firmware configures this cell via a mode register.
Output modes
There are two fundamentally different ways an MCU pin can drive a digital output signal: push-pull and open-drain. The distinction matters whenever multiple devices share a wire.
and low. Two devices can't share a wire — one drives high while other drives low → short. Open-drain VDD (external) R PIN N DATA_OUT Drives low only. Floats high via R. Multiple devices can share one wire — any device can pull low, none fights another.
Push-pull uses both P- and N-FETs; open-drain uses only an N-FET with an external pull-up resistor.
Push-pull uses both a P-channel FET (source to VDD) and an N-channel FET (source to GND), driven by complementary signals from the output data register. When DATA_OUT = 1, the P-FET is on and the N-FET is off — the pin is actively pulled to VDD. When DATA_OUT = 0, the N-FET is on — the pin is actively pulled to GND. There is never a floating state; the pin is always driven. This is fast, low-impedance, and the right choice for most digital outputs: LEDs, logic levels, clock lines that don't need wired-AND.
Open-drain uses only the N-FET. When DATA_OUT = 0, the FET turns on and pulls the pin to GND. When DATA_OUT = 1, the FET is off and the pin floats — its voltage is set entirely by whatever external resistor or device is connected. This has a critical property: two or more open-drain devices can share a single wire without any risk of bus contention. If one device drives low and another drives low simultaneously, both are just parallel N-FETs to GND — no short circuit. The line only goes high when all devices release it and the pull-up resistor lifts it.
I2C requires open-drain. SDA and SCL on an I2C bus are both open-drain lines with pull-up resistors. Any device on the bus can pull a line low (to hold the bus or signal NAK) without knowing what other devices are doing. If any device were push-pull, it could fight a device trying to drive the opposite value. The wired-AND behaviour of open-drain is not a limitation — it is the feature that makes multi-master I2C arbitration possible.
Input modes
On the receive side, the pad cell connects the physical pin to a digital input buffer. The mode register controls whether an internal pull resistor is enabled and in which direction.
Internal pull resistors (typically 40–100 kΩ) prevent floating inputs from picking up noise when no external signal is connected.
A floating input has no defined default state. When nothing is connected, the pin is a tiny antenna — capacitive coupling to adjacent traces can drive it anywhere between GND and VDD. A floating GPIO pin will read random values. Always configure a pull direction on any pin that can be left undriven.
Pull-up connects an internal resistor (typically 40–100 kΩ) from the pin to VDD. The pin defaults to logic HIGH. A button or open-drain device connected to GND overrides the pull; when it releases, the resistor restores the high state. Active-low buttons work with pull-ups — when pressed, the button shorts the pin to GND (logic 0); released, the resistor pulls it back to VDD (logic 1).
Pull-down is the mirror: resistor to GND, default LOW. Active-high buttons (to VDD) work with pull-downs.
Schmitt trigger hysteresis
A standard digital buffer switches at a single threshold. If the input signal is slow (e.g., an RC-filtered signal or a long trace with reflections), it spends time near the threshold, causing the output to oscillate rapidly — a burst of spurious transitions known as glitching. The Schmitt trigger fixes this with hysteresis: two separate thresholds, one for rising edges and one for falling edges.
Once the input crosses VH the output latches HIGH and ignores noise until the input falls below the separate VL threshold.
On the EFR32, all GPIO input modes include a Schmitt trigger by default. An additional glitch filter (2-cycle deglitcher) is available by selecting gpioModeInputPullFilter or gpioModeWiredAndPullUpFilter — useful for slow signals like mechanical buttons where even a few microseconds of bounce would register as a false edge.
EFR32 GPIO API
The EFR32 has up to six GPIO ports (PA–PF on xG12/13; PA–PD on xG21/22), each with up to 16 pins. All GPIO configuration goes through a single function: 
void GPIO_PinModeSet(GPIO_Port_TypeDef port,
                     unsigned int      pin,
                     GPIO_Mode_TypeDef mode,
                     unsigned int      out);
The out parameter sets the initial value of the output data register and, for input modes, selects pull direction: out=1 enables pull-up, out=0 enables pull-down.
Mode constant Direction Description
gpioModeDisabled—High-impedance, output disabled
gpioModeInputInFloating input, Schmitt trigger
gpioModeInputPullInInput with pull-up (out=1) or pull-down (out=0), Schmitt trigger
gpioModeInputPullFilterInPull + Schmitt + 2-cycle glitch filter
gpioModePushPullOutPush-pull output, standard drive
gpioModePushPullAlternateOutPush-pull with alternate drive strength setting
gpioModeWiredAndOutOpen-drain (wired-AND): drives low, floats high
gpioModeWiredAndPullUpOutOpen-drain with internal pull-up (I2C style)
gpioModeWiredAndPullUpFilterOutOpen-drain + pull-up + glitch filter
gpioModeWiredOrOutOpen-source (wired-OR): drives high, floats low
 
#include "em_gpio.h"
#include "em_cmu.h"

void gpio_init(void) {
    CMU_ClockEnable(cmuClock_GPIO, true);

    // LED on PF4 — push-pull output, start low (off)
    GPIO_PinModeSet(gpioPortF, 4, gpioModePushPull, 0);

    // Button on PF6 — input with internal pull-up (active low)
    GPIO_PinModeSet(gpioPortF, 6, gpioModeInputPull, 1);

    // I2C SDA on PD14 — open-drain with pull-up (wired-AND)
    GPIO_PinModeSet(gpioPortD, 14, gpioModeWiredAndPullUp, 1);
}

// Output manipulation
void led_on(void)     { GPIO_PinOutSet(gpioPortF, 4); }
void led_off(void)    { GPIO_PinOutClear(gpioPortF, 4); }
void led_toggle(void) { GPIO_PinOutToggle(gpioPortF, 4); }

// Input read — returns 0 (pressed) or 1 (released) for active-low button
bool button_pressed(void) {
    return GPIO_PinInGet(gpioPortF, 6) == 0;
}
Drive strength
EFR32 output pins have two drive-strength settings, each with a "main mode" and an "alternate mode" (for when the pin is used as an alternate function): 
// High drive for FET gates, buzzers, LEDs without external buffer
GPIO_DriveStrengthSet(gpioPortF,
                      gpioDriveStrengthStrongAlternateStrong);

// Low drive (default) for ordinary logic signals — less EMI
GPIO_DriveStrengthSet(gpioPortF,
                      gpioDriveStrengthWeakAlternateWeak);
Higher drive strength = faster edges = more EMI. Use the minimum drive strength that still meets your rise/fall time requirements, especially near RF sections. On the EFR32, the default (weak) mode is adequate for 3.3 V logic; strong mode is needed when driving capacitive loads quickly (e.g., a high-speed SPI clock with heavy trace capacitance).
Alternate functions and pin routing
Every MCU pin can serve multiple functions. The same physical pad that works as a GPIO output can double as the TX pin of USART0, the CLK of SPI0, the CC0 capture pin of TIMER1, or the positive input of ADC0 — you choose by configuring the pin's alternate function and telling the peripheral which pin location to use.
The EFR32 has a flexible routing scheme rather than a fixed pinout table. Most peripherals offer several locations — different physical pin assignments — so you can move a peripheral to avoid PCB routing conflicts.
One physical pin routed to GPIO or one of several peripheral functions. Only one can be active at a time.
Series 1 routing (LOC + ROUTEPEN)
On EFR32 Series 1, each peripheral has a ROUTELOC0 register (and ROUTELOC1 for peripherals with more pins) that selects which numbered location each signal appears on, and a ROUTEPEN register that enables each routed pin: 
// Route USART0 TX/RX to location 0 (PA0/PA1 on xG12)
// and enable those pins in the USART0 peripheral
GPIO_PinModeSet(gpioPortA, 0, gpioModePushPull, 1);  // TX
GPIO_PinModeSet(gpioPortA, 1, gpioModeInputPull,  1);  // RX

USART0->ROUTELOC0 = USART_ROUTELOC0_TXLOC_LOC0
                  | USART_ROUTELOC0_RXLOC_LOC0;
USART0->ROUTEPEN  = USART_ROUTEPEN_TXPEN
                  | USART_ROUTEPEN_RXPEN;
Series 2 routing (GPIO route registers)
On EFR32 Series 2, the routing registers moved into the GPIO peripheral. You specify the exact port and pin number rather than a location index, which eliminates the need to look up which location maps to which pin: 
// Route EUSART0 TX/RX to PA0/PA1 on Series 2
GPIO_PinModeSet(gpioPortA, 0, gpioModePushPull, 1);
GPIO_PinModeSet(gpioPortA, 1, gpioModeInputPull, 1);

// Set port and pin for each signal
GPIO->USARTROUTE[0].TXROUTE =
    (gpioPortA << _GPIO_USART_TXROUTE_PORT_SHIFT) |
    (0 << _GPIO_USART_TXROUTE_PIN_SHIFT);

GPIO->USARTROUTE[0].RXROUTE =
    (gpioPortA << _GPIO_USART_RXROUTE_PORT_SHIFT) |
    (1 << _GPIO_USART_RXROUTE_PIN_SHIFT);

// Enable the routed pins
GPIO->USARTROUTE[0].ROUTEEN =
    GPIO_USART_ROUTEEN_TXPEN | GPIO_USART_ROUTEEN_RXPEN;
Always configure the GPIO pin mode before enabling the peripheral routing. The peripheral takes over the pad when routing is enabled; if the pad is still in disabled mode at that point, the peripheral output may be undefined for one clock cycle. Also, if a pin is used as an analog input (ADC), set its mode to gpioModeDisabled — the analog input path bypasses the digital input buffer, and leaving the digital buffer enabled wastes current and adds noise.
GPIO interrupts
Any GPIO pin can generate an external interrupt on a rising edge, falling edge, or both. The EFR32 routes GPIO interrupts through two NVIC vectors: GPIO_EVEN_IRQn for even-numbered pins (0, 2, 4, …, 14) and GPIO_ODD_IRQn for odd-numbered pins (1, 3, 5, …, 15). Inside the handler, you read GPIO_IntGet() to find which pin fired. 
#include "em_gpio.h"

void gpio_interrupt_init(void) {
    CMU_ClockEnable(cmuClock_GPIO, true);

    // Button on PF6 with internal pull-up — active low
    GPIO_PinModeSet(gpioPortF, 6, gpioModeInputPull, 1);

    // Interrupt on falling edge (button press), rising edge (release), enabled
    GPIO_ExtIntConfig(gpioPortF,
                      6,
                      6,       // interrupt number (0–15, must match pin)
                      false,   // rising edge — no
                      true,    // falling edge — yes
                      true);   // enable immediately

    NVIC_ClearPendingIRQ(GPIO_EVEN_IRQn);
    NVIC_EnableIRQ(GPIO_EVEN_IRQn);  // pin 6 is even
}

void GPIO_EVEN_IRQHandler(void) {
    uint32_t flags = GPIO_IntGet();
    GPIO_IntClear(flags);

    if (flags & (1u << 6)) {
        // PF6 fell → button pressed
        handle_button_press();
    }
}
One interrupt number per pin number, not per port. The EFR32 GPIO interrupt controller assigns interrupt numbers 0–15 globally, one per pin index. You cannot simultaneously have interrupts on PA4 and PB4 — both would use interrupt number 4. Choose which port to use for each interrupt number, and assign only one port's pin per number.
Debouncing mechanical contacts
Mechanical switches don't produce a clean single transition when pressed. The metal contacts bounce off each other for anywhere from a few hundred microseconds to tens of milliseconds, producing a burst of rapid transitions before settling to the final state. Without debouncing, one button press generates dozens of GPIO interrupt events.
Hardware debounce: an RC low-pass filter (e.g., 10 kΩ + 100 nF → τ = 1 ms) smooths the bounce, and the Schmitt trigger on the GPIO input reconstructs a clean edge from the filtered waveform. This is reliable and costs almost nothing in firmware, but requires two passives on the PCB.
Software debounce: on the first edge interrupt, disable the GPIO interrupt and start a timer for ~20 ms. When the timer fires, re-read the pin: if it is still in the expected pressed state, accept the press; then re-enable the interrupt. This is the right choice when PCB real estate is tight. 
// Software debounce using a one-shot timer (pseudo-code)
static volatile bool debounce_active = false;

void GPIO_EVEN_IRQHandler(void) {
    GPIO_IntClear(GPIO_IntGet());
    if (!debounce_active) {
        debounce_active = true;
        GPIO_IntDisable(1u << 6);        // silence pin 6 interrupts
        TIMER_start_oneshot_ms(20);       // 20 ms settle time
    }
}

void TIMER_debounce_callback(void) {
    debounce_active = false;
    if (!GPIO_PinInGet(gpioPortF, 6)) {  // still low (pressed)?
        handle_button_press();
    }
    GPIO_IntEnable(1u << 6);            // re-arm for next press
}
