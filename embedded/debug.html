15 · Debugging & Test
SWD · JTAG · OpenOCD · GDB · Fault Analysis · Logic Analyzer · Oscilloscope
Embedded debugging differs from software debugging on a PC in one fundamental way: the system under test is a physical object, and the bugs are often at the intersection of software and hardware. A wrong register write crashes the MCU. A timing bug only appears at a particular CPU frequency. A floating input pin causes sporadic failures. Understanding the full debug toolchain — from the hardware debug port on the chip to the waveforms on the oscilloscope — is what separates productive firmware engineers from people who guess and reflash.
SWD & JTAG
Both SWD and JTAG are physical interfaces that give a host computer access to the chip's internal debug infrastructure without using any application-level resources (no UART pin, no CPU cycle budget during halted debug).
JTAG (IEEE 1149.1) uses four signals: TCK (clock), TMS (mode select), TDI (data in), TDO (data out), plus an optional TRST reset. It is a daisy-chain bus — multiple devices on the same board can share the JTAG chain.
SWD (Serial Wire Debug) is ARM's two-wire alternative: SWDIO (bidirectional data) and SWDCLK (clock). It provides the same debug and memory-access capability as JTAG in two fewer pins — important on small packages. Nearly all Cortex-M development uses SWD. The EFR32 exposes SWD on dedicated pins that remain functional regardless of GPIO configuration.
Cortex-M Debug Hardware
Every Cortex-M core contains a standardised set of debug components accessible via the Debug Access Port (DAP). You rarely configure these directly — OpenOCD and GDB do it for you — but knowing they exist explains what the debugger can and cannot do.
DAP — Debug Access Port. The entry point accessible over SWD or JTAG. Contains the SW-DP (Serial Wire Debug Port) or JTAG-DP that accepts commands from the probe, plus one or more Access Ports (APs) for different subsystems.
AHB-AP — AHB Access Port. Translates DAP commands into bus transactions on the main AHB bus. This is what lets GDB read and write any memory address — RAM, Flash, and memory-mapped peripheral registers — even while the CPU is running. It is the foundation of non-intrusive debug (reading a variable without halting).
FPB — Flash Patch and Breakpoint. Provides up to 8 hardware breakpoints (6 instruction, 2 literal on Cortex-M3/M4). A hardware breakpoint halts the CPU the instant the PC reaches a target address, with zero code modification. This is different from software breakpoints, which replace the instruction at the target with a BKPT opcode — software breakpoints cannot be set in execute-only Flash on some MCUs.
DWT — Data Watchpoint and Trace. Provides hardware watchpoints (halt on a specific memory address being read or written) and the cycle counter (DWT->CYCCNT) — a 32-bit register that increments every CPU clock cycle. Reading CYCCNT before and after a code section gives exact cycle-accurate timing without any timer peripheral. 
// Enable the DWT cycle counter (must be done explicitly)
CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
DWT->CYCCNT = 0;
DWT->CTRL   |= DWT_CTRL_CYCCNTENA_Msk;

// --- Time a section ---
uint32_t t0 = DWT->CYCCNT;
do_something();
uint32_t cycles = DWT->CYCCNT - t0;
// cycles / SystemCoreClock gives time in seconds
ITM — Instrumentation Trace Macrocell. A dedicated output channel for trace data. By writing to ITM stimulus ports, firmware can emit printf-style messages over the SWO (Serial Wire Output) pin without using a UART. See printf & ITM tracing below.
OpenOCD & GDB
OpenOCD (Open On-Chip Debugger) is open-source software that runs on the host PC. It speaks USB to the debug probe (J-Link, ST-Link, CMSIS-DAP) on one side, and exposes a GDB server (port 3333 by default) on the other. GDB connects to this server and sends standard remote serial protocol commands; OpenOCD translates them into SWD/JTAG transactions. 
# Start OpenOCD for EFR32 with a J-Link probe
openocd -f interface/jlink.cfg \
        -f target/efm32.cfg

# GDB connects to the running OpenOCD server
arm-none-eabi-gdb firmware.elf

(gdb) target remote localhost:3333   # connect to OpenOCD
(gdb) monitor reset halt             # reset and halt CPU
(gdb) load                           # flash firmware.elf
(gdb) monitor reset halt             # reset again after flash
(gdb) continue                       # run
Essential GDB commands for embedded work: 
# Execution control
(gdb) break main              # set breakpoint at function
(gdb) break uart.c:47         # set breakpoint at file:line
(gdb) watch g_rx_count        # hardware watchpoint: halt on write
(gdb) continue / c            # resume execution
(gdb) step / s                # step one C source line (enters calls)
(gdb) next / n                # step over (doesn't enter calls)
(gdb) finish                  # run until current function returns
(gdb) stepi / si              # step one machine instruction

# Inspect state
(gdb) info registers          # all CPU registers
(gdb) print g_sensor_val      # print variable value
(gdb) print/x 0x40010000      # print memory address in hex
(gdb) x/16xw 0x20000000       # examine 16 words from SRAM base
(gdb) backtrace / bt          # call stack

# Memory
(gdb) set var g_mode = 2      # write variable while running
(gdb) monitor mdw 0x40000000  # raw word read via OpenOCD

# Reset
(gdb) monitor reset halt      # hardware reset, halt at reset vector
Debug build flags matter. Compile with -g3 -O0 during development: -g3 includes full debug symbols (variable names, types, line numbers), -O0 disables optimisation so variables aren't optimised away and single-stepping follows the source line-by-line. Release builds use -O2 or -Os which makes GDB nearly unusable — variables appear as <optimized out> and stepping jumps seemingly random lines.
Fault Analysis
When firmware accesses an invalid memory address, executes an undefined instruction, or violates a memory protection boundary, the Cortex-M processor takes a fault exception. The most common is HardFault. Without a debugger attached, this causes an immediate reset and all state is lost. The right approach is to install a fault handler that captures state before resetting.
The key registers to read on any fault: 
// Read these in your HardFault_Handler to diagnose the fault

// Configurable Fault Status Register — which fault and why
uint32_t CFSR  = SCB->CFSR;

// Individual sub-registers within CFSR
uint8_t  MMFSR = (CFSR >> 0)  & 0xFF;  // MemManage faults
uint8_t  BFSR  = (CFSR >> 8)  & 0xFF;  // BusFault faults
uint16_t UFSR  = (CFSR >> 16) & 0xFFFF; // UsageFault faults

// Fault addresses (only valid when corresponding address-valid bit set)
uint32_t MMFAR = SCB->MMFAR;  // MemManage fault address
uint32_t BFAR  = SCB->BFAR;   // BusFault address

// Hard Fault Status Register
uint32_t HFSR  = SCB->HFSR;
The most useful diagnostic pattern is to read the stacked PC — the address of the instruction that caused the fault — from the exception stack frame: 
// In startup.s or a C wrapper:
// On exception entry, Cortex-M hardware pushes r0-r3, r12, LR, PC, xPSR
// to the process stack (PSP) if in thread mode, or main stack (MSP) in handler mode.

void HardFault_Handler(void) {
    __asm volatile (
        "tst    lr, #4          \n"   // test bit 2 of LR (EXC_RETURN)
        "ite    eq              \n"   // if EQ (bit 2 clear): came from MSP
        "mrseq  r0, msp         \n"   //   r0 = MSP
        "mrsne  r0, psp         \n"   //   r0 = PSP
        "b      hard_fault_c    \n"   // call C handler with stack frame ptr
    );
}

void hard_fault_c(uint32_t *frame) {
    uint32_t stacked_pc = frame[6];  // PC is 7th word pushed
    uint32_t stacked_lr = frame[5];
    uint32_t stacked_r0 = frame[0];

    // Log or breakpoint here
    (void)stacked_pc;  // inspect in debugger
    for (;;) {}        // halt — don't reset silently
}
With a debugger attached, a simpler approach: set a breakpoint inside HardFault_Handler, then use GDB's backtrace after the fault. If GDB doesn't automatically unwind through the exception frame, use x/8xw $msp (or $psp) to read the pushed frame manually and identify the faulting PC.
Common HardFault causes: (1) NULL pointer dereference — stacked PC points just after the bad load/store. (2) Stack overflow — stacked frame lands in invalid memory; BFSR UNSTKERR bit set. (3) Unaligned access with strict alignment required — UFSR UNALIGNED bit set. (4) Division by zero with UFSR trap enabled — UFSR DIVBYZERO bit set. (5) Executing data as code — bad function pointer, usually UFSR INVSTATE.
printf & ITM Tracing
The simplest debug technique is inserting printf calls and observing the output. There are two output paths available on Cortex-M:
UART printf. Retarget the C library's _write() system call to send bytes over a UART. Simple, widely understood, works on any toolchain. Disadvantage: consumes a UART peripheral and GPIO pins, adds UART baud rate latency, and transmitting inside a time-critical ISR will distort timing.
ITM/SWO printf (non-intrusive). Send characters through the ITM Stimulus Port 0, which the debug probe captures on the SWO pin and forwards to OpenOCD's trace log. No UART needed, extremely fast (runs at tens of MHz), and the output appears in the debug host without touching any application peripheral. Disadvantage: requires a probe that supports SWO (J-Link does; cheaper CMSIS-DAP adapters often do not). 
// ITM_SendChar — retarget fputc to ITM stimulus port 0
int fputc(int ch, FILE *f) {
    (void)f;
    // Wait until port 0 is not busy
    while (ITM->PORT[0].u32 == 0) {}
    ITM->PORT[0].u8 = (uint8_t)ch;
    return ch;
}

// OpenOCD configuration to capture SWO:
// monitor tpiu config internal swo.log uart false 38400000 2000000
// Then: tail -f swo.log   in a terminal
Semihosting is a third option: ARM's mechanism for routing printf output through the debug connection using a special BKPT instruction. It works without any pins, but is extremely slow (each character halts the CPU and waits for the host to respond) and completely breaks without a debugger attached. Use it only for early bringup on development hardware where the debugger is always connected.
Logic Analyzer
A logic analyzer captures multiple digital signals simultaneously and displays them as a waveform timeline. Unlike an oscilloscope it records logic levels (1/0), not voltage, and can capture hours of traffic at megahertz sample rates. The most important feature for embedded work is protocol decoding — the analyser understands UART, SPI, I2C, CAN, USB, and dozens of other protocols and annotates the waveform with decoded bytes and fields.
Sigrok / PulseView is the open-source analyser software; it works with cheap ($10–30) USB logic analysers (Saleae clones based on the Cypress FX2) up to professional units. For EFR32 work, a 4–8 channel analyser at 24 MHz sample rate is sufficient for most protocols.
Key setups for common protocols:
  • UART decode: Attach one probe to TX and/or RX. In PulseView, add the UART decoder, set baud rate to match firmware (configTICK_RATE_HZ is irrelevant — use the USART baud rate), set data bits/parity/stop to match. Decoded bytes appear above the waveform. Look for framing errors (STOP bit low) or unexpected bytes to diagnose mis-configured baud generators.
  • SPI decode: Attach probes to SCLK, MOSI, MISO, CS. Set CPOL and CPHA in the decoder to match firmware. The decoder shows each byte per CS assertion. Common issue to check: is CS actually being deasserted between transfers? Some bugs only manifest as two CS pulses that merge because the MCU didn't deassert long enough.
  • I2C decode: Attach probes to SCL and SDA. The decoder shows each address and data byte, marks ACK/NAK, and flags protocol violations. A NAK in an unexpected place immediately shows whether the device acknowledged its address — if not, check pull-up values and device power.
Triggering. Logic analysers can trigger (start recording) on a specific bit pattern, protocol event, or signal edge. Use triggers to capture rare events — e.g., "start recording only when this CAN message ID appears" or "capture only on SPI CS assertion." Without triggers you drown in data.
Oscilloscope Techniques
The oscilloscope shows voltage vs time. Where a logic analyser tells you what the signal says, the oscilloscope tells you how well it says it. Signal integrity problems, overshoot, ringing, ground noise, and incorrect voltage levels are invisible to a logic analyser but obvious on the oscilloscope.
Key measurements for embedded work:
  • Rise/fall time. The time for a signal to transition from 10% to 90% of its swing. For SPI or I2C at high speeds, slow edges (caused by excessive bus capacitance or weak pull-ups) cause setup/hold violations. A 400 kHz I2C bus should have rise times under ~300 ns.
  • Overshoot and ringing. Inductive transmission lines ring when driven hard. Excessive ringing can exceed Schmitt trigger hysteresis thresholds and cause multiple logic transitions per intended edge. Add series termination resistors (22–33 Ω) near the driver to damp ringing.
  • Ground noise. Use the scope's math channel to subtract two channels: CH1 on a signal, CH2 on the nearby ground plane. Any difference is conducted ground noise. High ground noise usually points to a missing or mis-placed decoupling capacitor.
  • PWM duty cycle verification. Trigger on the rising edge, measure the pulse width and period. Compare with the expected value from the timer register settings. If they disagree, check clock tree configuration or prescaler values.
  • Timing loops. Toggle a GPIO at the start and end of a critical section, then measure the resulting pulse on the scope. Faster and lower-overhead than CYCCNT for verifying ISR latency or DMA trigger-to-transfer timing.
Probe ground matters. The oscilloscope probe's ground clip must be connected to a ground point as close to the signal under test as possible. A long ground clip acts as an inductor, adding ringing and noise to every measurement — the waveform looks broken when the probing technique is the problem.
