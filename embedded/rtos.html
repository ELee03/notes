14 · RTOS Concepts
Tasks · Scheduling · Semaphores · Mutexes · Queues · Priority Inversion
A real-time operating system (RTOS) is a thin software layer that adds one crucial capability the bare-metal superloop lacks: the ability to run several logically independent activities concurrently, each with its own timing guarantee. It does this not with multiple processors, but by rapidly switching the single CPU between tasks — fast enough that each task appears to run continuously. The "real-time" in the name means the kernel provides deterministic scheduling with bounded latency, not that it is particularly fast.
Why an RTOS?
The superloop (Lesson 13) works well when all tasks are quick and similarly timed. It falls apart when:
  • Timing requirements diverge. Blinking an LED every second and servicing a 1 Mbit/s CAN bus are both "things the firmware does," but one needs microsecond response and the other is happy with milliseconds. A single loop iteration cannot satisfy both without compromising one.
  • Long operations block everything. If one part of the system needs to wait for an SD card write (tens of milliseconds), the entire superloop stalls. There is no way to keep servicing other work during the wait.
  • Complexity grows non-linearly. Each new feature added to a superloop must interleave cleanly with every existing feature. An RTOS lets each concern live in its own task, with explicit synchronization at the boundaries.
An RTOS solves these problems by decomposing the firmware into tasks — independent threads of execution — and providing a scheduler that decides which task runs at each moment, plus synchronization primitives (semaphores, mutexes, queues) for safe communication between tasks.
RTOS vs bare-metal is a tradeoff, not an upgrade. An RTOS adds code size (~5–20 KB for FreeRTOS), a stack per task (often 512–2048 bytes each), and non-determinism at context-switch boundaries. For a simple sensor node doing one thing, the superloop is the right answer. Use an RTOS when you have genuinely concurrent activities with independent timing requirements.
Tasks & the Kernel
A task (also called a thread) is a C function that looks like it runs forever. The RTOS gives it an exclusive, private stack and assigns it a priority. From the task's perspective it has the whole CPU to itself; the RTOS invisibly time-shares the hardware. 
void sensor_task(void *pvParameters) {
    // Task body — runs as if it owns the CPU
    for (;;) {
        uint16_t val = ADC_read();
        xQueueSend(sensor_queue, &val, portMAX_DELAY);
        vTaskDelay(pdMS_TO_TICKS(10));  // sleep 10 ms
    }
    // Tasks should never return. If they do, behaviour is undefined.
}
Every task is in exactly one state at any moment. The state diagram below covers all four states the kernel tracks:

Running — exactly one task (or the idle task) occupies the CPU at a time. Ready — the task could run but a higher-priority task is currently running. Blocked — the task is waiting for something: a semaphore, a queue item, a timeout. It will not consume CPU while blocked. Suspended — explicitly paused via vTaskSuspend(); not considered for scheduling until vTaskResume() is called.
Each task has a private stack — a region of RAM that the kernel switches in when that task runs. The stack holds the task's local variables and the saved CPU context (registers) when the task is not running. Sizing stacks correctly is one of the most error-prone parts of RTOS development (see Stack Overflow).
Scheduler & Context Switch
The scheduler decides which task runs. FreeRTOS and most embedded RTOSes use a preemptive, priority-based scheduler with round-robin for equal-priority tasks.
The rules are simple:
  • The highest-priority ready task always runs.
  • If two ready tasks share the same priority, they take turns each tick (round-robin).
  • A running task is immediately preempted if a higher-priority task becomes ready (e.g., because an ISR unblocked it).
The tick is a periodic timer interrupt that drives the scheduler. FreeRTOS defaults to 1 ms ticks (1000 Hz). Each tick the scheduler checks whether a context switch is needed. The tick period sets the granularity of vTaskDelay() — you cannot sleep for less than one tick.
Preemptive scheduling: T1 (highest priority) interrupts T2 and T3 the moment it becomes ready. Lower-priority tasks only run when T1 blocks.
A context switch is the mechanism that makes this possible. When the scheduler preempts a running task, it must save the task's complete CPU state so it can resume later as if nothing happened. On Cortex-M, that means:
  • Hardware-saved automatically (on any exception entry): r0–r3, r12, LR, PC, xPSR — 8 registers pushed to the current stack.
  • Software-saved by the RTOS (in the PendSV handler): r4–r11 (and s16–s31 if FPU is active).
  • Stack pointer saved to the TCB (Task Control Block — the kernel's per-task data structure). The new task's SP is loaded from its TCB.
FreeRTOS triggers context switches via PendSV, the lowest-priority exception on Cortex-M. Using PendSV means the context switch always happens after all higher-priority exceptions have completed — it never preempts real hardware ISRs.
Cooperative vs preemptive scheduling. In a cooperative (non-preemptive) RTOS, a task runs until it explicitly yields by calling taskYIELD() or a blocking API. The kernel never forcibly switches away from a running task. Cooperative scheduling makes reasoning about shared state easier but breaks the real-time guarantee — a task that spins never gives others a chance to run. Most real-time systems use preemptive scheduling for this reason.
Semaphores & Mutexes
Tasks need two things from each other: signals ("something happened") and shared resources ("only one of us uses this at a time"). The RTOS provides primitives for each.
Binary Semaphore
A binary semaphore is a flag with two operations: give (post/signal) and take (pend/wait). It has a count of 0 or 1. A task calling xSemaphoreTake() on a zero-count semaphore blocks until another task (or ISR) calls xSemaphoreGive().
The canonical use case is ISR → task signaling: the ISR runs to completion (quickly — no blocking allowed in ISRs), then gives the semaphore to wake the task that will do the heavy lifting. 
SemaphoreHandle_t uart_rx_sem;

// --- ISR (runs in interrupt context) ---
void UART0_IRQHandler(void) {
    BaseType_t higher_prio_woken = pdFALSE;

    ring_push(&rx_ring, UART0->RXDATA);  // store byte

    // Give semaphore FROM ISR — use the FromISR variant!
    xSemaphoreGiveFromISR(uart_rx_sem, &higher_prio_woken);

    // If giving the semaphore unblocked a higher-priority task,
    // request a context switch immediately on ISR exit.
    portYIELD_FROM_ISR(higher_prio_woken);
}

// --- Task (runs in task context) ---
void uart_rx_task(void *pvParameters) {
    for (;;) {
        // Block indefinitely until ISR gives the semaphore
        xSemaphoreTake(uart_rx_sem, portMAX_DELAY);
        // Process ring buffer here...
        process_incoming_data();
    }
}
FromISR variants are mandatory in ISRs. FreeRTOS maintains a critical section internally. Calling xSemaphoreGive() from an ISR corrupts kernel state. Any FreeRTOS API called from interrupt context must be the ...FromISR version. The pxHigherPriorityTaskWoken pattern ensures the context switch happens on ISR exit, not mid-ISR.
Mutex
A mutex (mutual exclusion) is a semaphore with ownership. Unlike a binary semaphore, only the task that took the mutex can give it back. Mutexes protect shared resources — peripheral registers, global data structures, external flash — where only one task should operate at a time. 
MutexHandle_t spi_mutex;

void task_a(void *p) {
    for (;;) {
        xSemaphoreTake(spi_mutex, portMAX_DELAY);  // acquire
        spi_write(CMD_SENSOR_READ);                 // exclusive SPI access
        xSemaphoreGive(spi_mutex);                  // release
        vTaskDelay(pdMS_TO_TICKS(50));
    }
}

void task_b(void *p) {
    for (;;) {
        xSemaphoreTake(spi_mutex, portMAX_DELAY);
        spi_write(CMD_DISPLAY_UPDATE);
        xSemaphoreGive(spi_mutex);
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}
FreeRTOS mutexes implement priority inheritance to mitigate priority inversion (see below). Binary semaphores do not — use a mutex whenever protecting a shared resource, reserve binary semaphores for signaling.
Queues
A queue is a FIFO buffer that the kernel manages. It carries data — not just signals — between tasks. The sender copies data into the queue; the receiver copies data out. Because the kernel owns the buffer, there is no shared-memory race condition. 
#define SENSOR_QUEUE_LEN  8

typedef struct {
    uint32_t timestamp_ms;
    int16_t  temperature_cdeg;   // in 0.01 °C units
} sensor_reading_t;

QueueHandle_t sensor_queue;

// Producer task
void sensor_task(void *p) {
    sensor_reading_t reading;
    for (;;) {
        reading.timestamp_ms     = xTaskGetTickCount();
        reading.temperature_cdeg = read_temp_sensor();

        // Block up to 10 ms if queue is full
        xQueueSend(sensor_queue, &reading, pdMS_TO_TICKS(10));
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

// Consumer task
void logger_task(void *p) {
    sensor_reading_t reading;
    for (;;) {
        // Block indefinitely until a reading is available
        if (xQueueReceive(sensor_queue, &reading, portMAX_DELAY) == pdTRUE) {
            log_to_flash(reading.timestamp_ms, reading.temperature_cdeg);
        }
    }
}
Queue length and item size are fixed at creation time. The kernel allocates the buffer statically or from the heap. Queues are the primary mechanism for transferring ownership of data between tasks with different rates — the producer and consumer can run at different speeds, with the queue absorbing bursts up to its depth.
Priority Inversion
Priority inversion is one of the most famous failure modes in RTOS programming. It occurs when a high-priority task is blocked waiting for a resource held by a low-priority task, while a medium-priority task preempts the low-priority task — meaning the medium-priority task effectively runs before the high-priority one.
LOW holds the mutex. HIGH tries to acquire it and blocks. MED (no involvement with the mutex) preempts LOW — so HIGH is stuck waiting for LOW to run, which it can't until MED yields.
The fix is priority inheritance. When a mutex-holding task blocks a higher-priority task, the kernel temporarily boosts the holder's priority to match the highest waiter. In the example above, LOW would be temporarily elevated to HIGH's priority, so MED cannot preempt it. Once LOW releases the mutex, its priority reverts.
FreeRTOS mutexes implement priority inheritance; binary semaphores do not. This is the practical reason to choose a mutex over a binary semaphore when protecting shared resources. The Mars Pathfinder mission (1997) suffered a famous priority inversion bug that caused repeated system resets because its VxWorks RTOS priority inheritance was disabled on one mutex.
Stack Overflow
Every task has a fixed-size stack allocated at creation time. If a task's call depth or local variable usage exceeds the stack, it silently overwrites adjacent memory — typically another task's stack or TCB — causing corrupted behaviour that may not manifest until much later. Stack overflow is one of the hardest RTOS bugs to diagnose.
FreeRTOS provides two detection methods: 
// In FreeRTOSConfig.h:
// Method 1 — stack watermark check on each context switch (cheap)
#define configCHECK_FOR_STACK_OVERFLOW  1

// Method 2 — fill stack with 0xA5 at creation, check pattern on switch
#define configCHECK_FOR_STACK_OVERFLOW  2

// Hook called on detection (you implement this):
void vApplicationStackOverflowHook(TaskHandle_t xTask, char *pcTaskName) {
    // Log the offending task name, then halt or reset
    fault_handler("Stack overflow in task: %s", pcTaskName);
    for (;;) {}
}
To determine how much stack a task actually uses, call uxTaskGetStackHighWaterMark(NULL) from within the task. It returns the minimum number of free stack words that have remained since the task started. A value near zero means the stack is dangerously undersized.
Stack sizing rules of thumb. Start with more than you think you need (e.g., 512 words = 2 KB on a 32-bit MCU). Enable watermark monitoring in development. Measure actual usage, then trim with margin. Calls to printf-family functions consume surprisingly large stack — typically 300–800 bytes. Any task that calls into a complex library needs extra caution.
FreeRTOS in Practice
FreeRTOS is the most widely deployed embedded RTOS. It is open source, well-documented, and supported natively by most MCU vendors including Silicon Labs. The EFR32 SDK ships FreeRTOS integration out of the box.
The minimum configuration to get a FreeRTOS application running: 
// 1. FreeRTOSConfig.h — tune to your MCU and RAM budget
#define configUSE_PREEMPTION             1
#define configCPU_CLOCK_HZ               ( 38400000UL )  // EFR32 at 38.4 MHz
#define configTICK_RATE_HZ               ( 1000 )        // 1 ms tick
#define configMAX_PRIORITIES             ( 5 )
#define configMINIMAL_STACK_SIZE         ( 128 )         // words
#define configTOTAL_HEAP_SIZE            ( 8192 )        // bytes
#define configUSE_MUTEXES                1
#define configUSE_COUNTING_SEMAPHORES    1
#define configUSE_QUEUE_SETS             0
#define configUSE_TIMERS                 0
#define configCHECK_FOR_STACK_OVERFLOW   2

// 2. main.c — create tasks, then start scheduler
int main(void) {
    hardware_init();

    sensor_queue = xQueueCreate(SENSOR_QUEUE_LEN, sizeof(sensor_reading_t));
    spi_mutex    = xSemaphoreCreateMutex();
    uart_rx_sem  = xSemaphoreCreateBinary();

    xTaskCreate(sensor_task,  "sensor",  256, NULL, 2, NULL);
    xTaskCreate(logger_task,  "logger",  512, NULL, 1, NULL);
    xTaskCreate(uart_rx_task, "uart_rx", 256, NULL, 3, NULL);

    vTaskStartScheduler();  // never returns

    for (;;) {}  // unreachable
}
A few practical notes on the EFR32 + FreeRTOS combination:
  • SysTick drives the tick. FreeRTOS configures SysTick to fire at configTICK_RATE_HZ. Do not reconfigure SysTick elsewhere in your application.
  • Interrupt priority boundaries. FreeRTOS uses configMAX_SYSCALL_INTERRUPT_PRIORITY to set the boundary between ISRs that may call FreeRTOS APIs (at or below this priority) and ISRs that may not (above it). The latter class of ISRs must complete without touching any kernel object.
  • Idle task & tickless idle. FreeRTOS creates an idle task automatically. You can hook into it via vApplicationIdleHook(). The tickless idle extension (configUSE_TICKLESS_IDLE) suppresses SysTick when all tasks are blocked, enabling the EFR32 to enter EM1 or EM2 between events — critical for battery-powered designs.
