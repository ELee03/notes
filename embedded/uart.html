8 · UART
Frame Format · Baud Rate · RS-232 · Flow Control · EFR32 USART
UART — Universal Asynchronous Receiver-Transmitter — is the simplest serial protocol in common use. It requires only two wires (TX and RX), has no clock line, and needs no addressing because it is inherently point-to-point. UART is ubiquitous in embedded systems for debug consoles, firmware update over serial, GPS modules, GSM modems, inter-processor communication, and any situation where a short cable connects two devices.
"Universal" refers to the configurable frame format — data bits, parity, stop bits can all be tuned. "Asynchronous" is the key property: transmitter and receiver share no clock line. They must independently agree on a baud rate before communication, and each bit is decoded by sampling the line at the agreed rate after synchronizing on the falling edge of a start bit.
Frame format
A UART frame consists of: one start bit, 5–9 data bits (almost always 8), an optional parity bit, and one or two stop bits. The line idles high (logic 1, typically VDD). The start bit is always logic 0 — the falling edge of the idle-to-start transition is what synchronises the receiver.
8N1 frame: 1 start bit (LOW), 8 data bits LSB-first, 1 stop bit (HIGH). Blue dots: receiver sampling points at bit centres.
Data bits are transmitted LSB first. The character 'A' (0x41 = 0b01000001) has its bit 0 (the 1) transmitted first, followed by bits 1–6 (all 0s), then bit 7 (0). The receiver reconstructs the byte by reversing this sequence. This LSB-first convention is universal in UART and worth memorising to avoid confusion when reading logic analyser traces.
The standard frame descriptor uses three numbers: data bits / parity / stop bits. 8N1 means 8 data bits, no parity, 1 stop bit — the default for nearly all modern UART communication. Other configurations are rare: 8E1 (even parity) and 7E1 (7-bit ASCII with even parity) appear in legacy industrial equipment; 8N2 (two stop bits) is used when the receiver needs extra time between frames.

Baud rate and clock error tolerance
Baud rate is the number of symbols transmitted per second. For UART, one symbol equals one bit, so baud rate equals bit rate. Both sides must be configured identically; there is no handshake to discover baud rate automatically. Common values: 9600, 19200, 38400, 57600, 115200 (the ubiquitous default), 230400, 460800, 921600, and 1000000 baud.
Because the receiver has no clock reference from the transmitter, it must recover bit timing from the data stream itself. Modern UART hardware uses 16× oversampling: the receiver samples the RX line 16 times per bit period and takes the majority at samples 7, 8, 9 (centred on the nominal bit). The receiver syncs its counter to the falling edge of the start bit; any accumulated clock-rate mismatch shows up as a growing phase error.
Max error before missampling ≈ (0.5 × 1-bit width) / (N-bit frame) = 1/(2N)
For 10-bit frame (8N1): max ≈ 5% → practical limit: ±2.5% each side
A 10-bit frame (1 start + 8 data + 1 stop) has 10 bit periods of accumulated drift from the start-bit sync. To land within the middle third of the last bit, cumulative drift must be <1/6 bit ≈ 1.7%. The EFR32 baud rate divider introduces <0.5% error at most standard baud rates with a 38.4 MHz clock (38.4 MHz / 115200 baud = exactly 333.3, so the 16× oversampling fractional divider handles the fraction). Keep combined transmitter + receiver error under 3%; under 1% is comfortable. 
// EFR32 baud rate calculation (EMLIB does this automatically):
// BRR = round(f_PCLK / (16 × baud)) - 1

// BRR = round(38400000 / (16 × 115200)) - 1 = round(20.833) - 1 = 21 - 1 = 20

RS-232 vs TTL levels
UART is a protocol, not a voltage standard. The same framing can ride on different electrical levels:
Standard Logic 0 (SPACE) Logic 1 (MARK) Notes
CMOS/TTL 3.3 V~0 V~3.3 VStandard on modern MCUs
CMOS/TTL 5 V~0 V~5 VLegacy 5 V MCUs, Arduino 5 V
RS-232+3 to +15 V−3 to −15 VInverted polarity! Needs transceiver IC (MAX232)
RS-485A−B > +200 mVA−B < −200 mVDifferential, multi-drop, up to 1.2 km
When connecting an MCU to a legacy RS-232 device (PC COM port, industrial sensor), you cannot connect the 3.3 V TX directly to the RS-232 RX pin — the voltage levels are incompatible and the polarity is inverted. Use a transceiver chip (MAX232, MAX3232, or equivalent) that level-shifts and inverts the signal. Modern debug probes and USB-UART adapters (CP2102, CH340G, FT232R) present TTL UART directly, which is what you usually want.
Check Rx pin tolerance before connecting 5 V TTL to a 3.3 V MCU. Most EFR32 GPIO pins are not 5 V tolerant. Connecting a 5 V device's TX line directly to the EFR32 RX pin will damage the input. Use a resistor divider (1 kΩ / 2 kΩ) or a logic-level converter.
Flow control
Flow control prevents the receiver from being overwhelmed when its buffer or processing cannot keep up with the transmitter. Without it, received bytes can overwrite each other in the FIFO (overrun error).
RTS/CTS hardware flow control. When the receiver's FIFO fills, it deasserts its RTS line; the sender sees CTS go inactive (after propagation delay) and pauses transmission.
Hardware flow control (RTS/CTS) uses two extra wires. The receiver's RTS (Request To Send) output signals whether it can accept data; this line connects to the transmitter's CTS (Clear To Send) input. When the receiver's buffer fills, it deasserts RTS, and the transmitter pauses after the current byte. When the buffer drains, RTS is reasserted and transmission resumes. The EFR32 USART hardware can manage this automatically — once configured, the hardware itself holds TX and monitors CTS with no software overhead.
Software flow control (XON/XOFF) uses in-band control characters: when the receiver sends 0x13 (Ctrl-S, XOFF), the transmitter stops; 0x11 (Ctrl-Q, XON) resumes it. This avoids extra wires but pollutes the data stream — XOFF/XON cannot appear in binary data. Use it only when the data is guaranteed to be printable ASCII and no extra wires are available.
EFR32 USART API
The EFR32 USART peripheral supports UART (asynchronous), SPI (synchronous), IrDA, and SmartCard modes from the same hardware. In UART mode, EMLIB's USART_InitAsync() configures all the relevant registers. 
#include "em_usart.h"
#include "em_cmu.h"
#include "em_gpio.h"

// ── Initialize USART0 for 115200 8N1 UART (Series 1) ──────────

void uart0_init(void) {
    CMU_ClockEnable(cmuClock_USART0, true);
    CMU_ClockEnable(cmuClock_GPIO,   true);

    // Configure GPIO pins before enabling routing
    GPIO_PinModeSet(gpioPortA, 0, gpioModePushPull, 1); // TX: push-pull, idle HIGH
    GPIO_PinModeSet(gpioPortA, 1, gpioModeInputPull, 1); // RX: input with pull-up

    USART_InitAsync_TypeDef init = USART_INITASYNC_DEFAULT;
    init.baudrate = 115200;
    init.databits = usartDatabits8;
    init.parity   = usartNoParity;
    init.stopbits = usartStopbits1;
    // For flow control: init.hwFlowControl = usartHwFlowControlCtsAndRts;
    USART_InitAsync(USART0, &init);

    // Route TX/RX to location 0 pins (check datasheet for your package)
    USART0->ROUTELOC0 = USART_ROUTELOC0_TXLOC_LOC0
                      | USART_ROUTELOC0_RXLOC_LOC0;
    USART0->ROUTEPEN  = USART_ROUTEPEN_TXPEN
                      | USART_ROUTEPEN_RXPEN;
}

// ── Polling transmit / receive ─────────────────────────────────

void uart_putchar(uint8_t c) {
    USART_Tx(USART0, c);          // blocks until TX buffer empty
}

void uart_puts(const char *s) {
    while (*s) uart_putchar(*s++);
}

uint8_t uart_getchar(void) {
    return USART_Rx(USART0);      // blocks until byte received
}

// ── Interrupt-driven receive with ring buffer ─────────────────

#include "ring_buffer.h"           // your ring buffer implementation
static ring_buf_t rx_ring;

void uart0_rx_irq_enable(void) {
    USART_IntClear(USART0, USART_IF_RXDATAV);
    USART_IntEnable(USART0, USART_IF_RXDATAV);  // RX data available
    NVIC_ClearPendingIRQ(USART0_RX_IRQn);
    NVIC_EnableIRQ(USART0_RX_IRQn);
}

void USART0_RX_IRQHandler(void) {
    // RXDATAV clears automatically when RXDATA is read
    uint8_t byte = USART_RxDataGet(USART0);
    ring_buf_push(&rx_ring, byte);
}
Series 2: EUSART. EFR32 Series 2 devices use EUSART0 (Enhanced USART) with driver em_eusart.h. The init structure changes to EUSART_UartInit_TypeDef and routing uses the GPIO peripheral's route registers (same pattern as GPIO Lesson 5). The functional behaviour — frame format, baud rate, oversampling — is identical.
Common failure modes
Error Cause EFR32 flag Fix
Framing error Stop bit not received as HIGH (wrong baud rate, line noise, mismatched frame format) USART_IF_FERR Verify baud rates match; check signal integrity with scope
Overrun error New byte arrived before firmware read the previous one from RXDATA USART_IF_RXOF Use interrupt-driven or DMA receive; add RTS/CTS flow control
Parity error Received parity bit does not match calculated parity (single-bit error in frame) USART_IF_PERR Verify parity setting matches; improve signal integrity (termination, shorter cable)
Break condition Line held LOW for >1 full frame time (intentional or wiring fault) USART_IF_FERR Intentional use: LIN bus wakeup, UART bootloader entry on some MCUs
Garbled data Clock mismatch, wrong data bits / stop bits setting, level mismatch (5V→3.3V) — Decode with logic analyser; verify full frame format agreement
The most common issue in practice is baud rate mismatch. If you see a flood of framing errors or the received bytes are consistently wrong, put a logic analyser on the line: measure the actual bit period of any known byte and compare to 1/baud_rate. A 4% mismatch at 115200 baud = 43.4 µs per bit vs expected 8.68 µs is immediately obvious on the analyser.
