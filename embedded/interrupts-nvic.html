Interrupts & the NVIC
Embedded Systems · Lesson 3
Why Interrupts?
A CPU can only do one thing at a time. A UART byte arrives in microseconds. A button press can happen at any moment. A timer fires every millisecond. How does a single CPU respond to all of these without missing any?
One answer is polling: the CPU loops continuously checking every peripheral flag in turn. It works but is wasteful — the CPU burns cycles asking "anything yet?" thousands of times for every event that actually occurs. It also introduces latency proportional to how many things are being polled; if the UART byte arrives while the CPU is busy checking the ADC, the byte may be overwritten in the hardware FIFO before the CPU gets back to read it.
The better answer is interrupts. The CPU runs normal code. When a peripheral needs attention, it asserts a hardware signal. The CPU detects this, finishes its current instruction, saves its context, jumps to a specific handler function, processes the event, restores context, and resumes exactly where it left off — all transparently, in hardware. Response latency is deterministic and small (12 cycles on Cortex-M4 for the fastest case); the CPU is free to do real work the rest of the time.
The Exception Model
On Cortex-M, the term exception covers everything that causes the processor to deviate from normal sequential execution — both hardware interrupts from peripherals and software-triggered events. Exceptions are numbered. The vector table (Lesson 1) maps each number to a handler address.
The first sixteen exception numbers are reserved by ARM for system-level events:
Number Name Priority Triggered by
1 Reset −3 (highest) Power-on, SYSRESETREQ, pin reset
2 NMI −2 Non-maskable interrupt (watchdog, voltage fault)
3 HardFault −1 Unhandled fault, escalated fault
4 MemManage Configurable MPU access violation
5 BusFault Configurable Invalid memory access on AHB
6 UsageFault Configurable Undefined instruction, divide-by-zero, unaligned
11 SVCall Configurable SVC instruction (RTOS syscalls)
14 PendSV Configurable Software-triggered; RTOS context switch
15 SysTick Configurable System tick timer overflow (RTOS heartbeat)
From exception 16 onward, all entries are IRQs — vendor-defined peripheral interrupts. Every chip defines its own set. On an EFR32BG22 there are around 50 IRQs: one per USART, one per GPIO port, one per timer, and so on. The Cortex-M4 supports up to 240 external IRQs; the M33 supports up to 480.
The NVIC
The Nested Vectored Interrupt Controller (NVIC) sits between peripheral interrupt signals and the CPU core. It is responsible for:
Enabling and disabling individual IRQs. A peripheral can signal an interrupt all it likes — if the NVIC has that IRQ disabled, the CPU never sees it. You enable an IRQ with NVIC_EnableIRQ(IRQn) and disable with NVIC_DisableIRQ(IRQn).
Tracking pending state. If an interrupt fires while the CPU is already in a higher-priority handler, the NVIC marks it as pending. The moment a higher-priority handler finishes (or the CPU priority drops), the pending interrupt is taken immediately. No events are dropped, even if the handler is temporarily busy.
Assigning priorities. Each IRQ has a programmable priority level. Higher-priority interrupts can preempt lower-priority ones. This is how a time-critical motor-control timer interrupt can interrupt a lower-priority UART receive handler mid-execution.
/* Enable USART0 receive interrupt at priority 3 */ NVIC_SetPriority(USART0_RX_IRQn, 3); NVIC_EnableIRQ(USART0_RX_IRQn); /* The handler — must match the name in the vector table */ void USART0_RX_IRQHandler(void) { uint8_t byte = USART0->RXDATA; /* reading clears the interrupt flag */ ring_buffer_push(&rx_buf, byte); }
Exception Entry and Exit
When the NVIC decides to take an exception, the core does the following automatically in hardware, before your handler runs:
The core saves R0–R3, R12, LR, PC, and xPSR onto the active stack — these are exactly the registers the C ABI designates as caller-saved, so the handler function can use them freely. The core then loads LR with a special EXC_RETURN value encoding which stack and mode to restore on exit, and jumps to the handler via the vector table.
Your handler runs as an ordinary C function. It reads the peripheral status register to find out what happened, processes the event, clears the interrupt flag in the peripheral, and returns normally. The BX LR at the end of the handler triggers the unstacking sequence: the hardware detects the special EXC_RETURN value in LR, pops the eight saved registers back from the stack, and resumes the interrupted code exactly where it left off.
The handler is just a C function. You do not need assembly or special return sequences. The ARM calling convention and the ABI-compiler agreement about caller/callee-saved registers means a normal function compiled with -mcortex-m4 works correctly as an exception handler. Only two things are non-negotiable: the handler name must match the vector table entry exactly (so the linker knows where to put the address), and the handler must clear the peripheral interrupt flag before returning, or it will fire again immediately.
Priorities and Preemption
The Cortex-M NVIC implements a priority scheme where lower numerical values mean higher priority. Priority 0 is the highest programmable priority; the fixed system exceptions (Reset, NMI, HardFault) have negative priority levels and always win.
The number of priority bits is implementation-defined — EFR32 Series 1 (M4F) implements 3 bits, giving 8 programmable levels (0–7). EFR32 Series 2 (M33) uses 4 bits for 16 levels. ARM requires priorities to be stored in the most-significant bits of the priority byte register, so you always write them left-aligned: priority 3 in a 3-bit implementation goes into bits [7:5] of the priority register, not bits [2:0].
Preemption is when a higher-priority exception interrupts a lower-priority handler mid-execution — exactly like an interrupt interrupting normal code, but between two handlers. The lower-priority handler is suspended, the higher-priority handler runs to completion, and then the lower-priority handler resumes. This nesting can be several levels deep.
Tail-chaining is an optimization: when a handler returns and there is another exception pending at any priority level, the core skips the unstacking/restacking sequence entirely and vectors directly to the next handler. This saves the ~12 cycles of a full exception entry/exit cycle. You get this for free — it happens transparently in hardware.
Writing Interrupt-Safe Code
The moment you introduce interrupts, you have concurrency. Your main-loop code and your interrupt handlers run on the same CPU but can interleave at any instruction boundary. A variable accessed from both contexts is a shared resource, and unprotected shared resources cause bugs that are intermittent, hard to reproduce, and impossible to find with a debugger.
The core problems:
Non-atomic read-modify-write. A 32-bit peripheral register that spans more than one bus transaction, or a multi-byte variable accessed with multiple instructions, can be partially updated when an interrupt fires between the read and the write.
/* UNSAFE: if an IRQ fires between the load and store, it may miss the update */ counter++; /* actually: LDREX, ADD, STREX on Cortex-M — but still a race if the IRQ reads counter between the load and the store */ /* SAFE: disable interrupts around the critical section */ __disable_irq(); /* sets PRIMASK = 1 */ counter++; __enable_irq(); /* clears PRIMASK */
The volatile keyword. Any variable modified inside an ISR and read in main-loop code (or vice-versa) must be declared volatile. Without it, the compiler is allowed to cache the value in a register and never re-read from memory — your main loop will read a stale value forever even though the ISR has updated the variable.
/* Correct: volatile tells the compiler this variable can change outside its view */ volatile uint32_t tick_count = 0; void SysTick_Handler(void) { tick_count++; } void main(void) { while (tick_count < 1000) { /* correct: re-reads tick_count from memory each iteration */ do_work(); } }
PRIMASK vs BASEPRI. __disable_irq() sets PRIMASK, which masks all exceptions with configurable priority — i.e., everything except Reset, NMI, and HardFault. This is the right choice for short critical sections. BASEPRI is more surgical: setting BASEPRI = 3 masks only exceptions with priority ≥ 3 while allowing higher-priority exceptions through. Use BASEPRI when you want latency-critical high-priority IRQs to still fire during a critical section.
Keep ISRs short. An ISR should do the minimum necessary — typically read the data register into a buffer and set a flag — then return. Anything that involves waiting, allocating memory, or long computation belongs in main-loop code that checks the flag. Long ISRs delay every other interrupt in the system, introducing latency jitter that can break timing-sensitive protocols.
