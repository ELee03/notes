13 · Bare-Metal Patterns
Startup · Superloop · State Machines · Ring Buffers · Driver Layers · volatile
"Bare metal" means firmware that runs directly on the hardware with no operating system beneath it. There is no scheduler, no process isolation, no memory allocator — just your code, the C runtime, and the hardware registers. This sounds limiting, but bare-metal firmware powers the vast majority of deployed embedded systems: medical sensors, motor controllers, wireless radios, automotive ECUs. The absence of an OS removes overhead, unpredictability, and attack surface.
The patterns in this lesson are the toolkit that makes bare-metal firmware manageable as it scales beyond a few hundred lines: how to structure the main loop, how to model system behaviour, how to move data safely between interrupt context and main context, and how to layer drivers so they remain testable.
Startup sequence
Before main() runs, the C runtime performs a fixed sequence of initialization steps. Understanding this matters when debugging startup crashes or when adding custom initialization (e.g. a watchdog that must be kicked before the clock initialization that takes 50 ms).
The C startup sequence runs before main(). Linker scripts define the boundaries of each section and the addresses used during the copy and zero operations.
Two consequences matter in practice. First, a global variable with an initialiser (e.g. int x = 42;) has its initial value stored in Flash and copied to RAM at startup — if your startup code is broken, x will be uninitialised garbage. Second, a static variable inside a function lives in .bss, is zeroed at startup, and retains its value across calls — it is not re-initialised each time the function is entered.
Superloop vs interrupt-driven architecture
Almost all bare-metal firmware is structured around one of two patterns, or a hybrid of both.
The superloop is the simplest structure: main() contains an infinite while(1) loop that polls every peripheral and processes any pending work on each iteration. Interrupts may be used only to set flags; the main loop checks those flags and does the actual work. 
// Superloop pattern
int main(void) {
    system_init();      // clocks, peripherals, GPIO

    while (1) {
        if (uart_rx_ready()) {
            process_uart_byte(uart_read());
        }
        if (adc_sample_ready()) {
            process_adc_sample(adc_read());
        }
        if (timer_elapsed()) {
            run_control_loop();
            timer_clear();
        }
        // sleep until next event (optional, saves power)
        __WFI();        // Wait For Interrupt — CPU halts until any interrupt
    }
}
The superloop is predictable and easy to debug — the execution path is always visible in the call stack. Its weakness is latency: if process_uart_byte() takes 2 ms and a time-critical ADC sample arrives during that window, it will be missed. The loop period is the sum of all handler times, which grows as the system gains features.
Interrupt-driven firmware does the real work inside interrupt handlers rather than polling from the main loop. The main loop may be empty (while(1) { __WFI(); }) or handle only low-priority background tasks. This gives better latency for high-priority events but introduces concurrency hazards: shared data accessed from both interrupt and main context must be protected.
In practice most systems use a hybrid: time-critical paths (UART RX byte, ADC sample) are handled in interrupt context; longer processing (packet parsing, state transitions, display updates) is done in the main loop, signalled by a flag set in the interrupt handler.
State machines
A finite state machine (FSM) is the standard way to model any behaviour that depends on history — a UART packet parser, a button debouncer, a connection state tracker, a motor sequence controller. The core insight is that what the system does next depends not just on the current input, but on what state it is in.
The idiomatic C implementation uses an enum for states and a switch statement for transitions: 
// Example: UART packet parser FSM
// Protocol: 0xAA (header), 1-byte length, N data bytes, 1-byte checksum

typedef enum {
    STATE_IDLE,
    STATE_GOT_HEADER,
    STATE_RECEIVING_DATA,
    STATE_CHECKSUM,
} parser_state_t;

static parser_state_t state = STATE_IDLE;
static uint8_t  expected_len;
static uint8_t  byte_count;
static uint8_t  rx_buf[64];
static uint8_t  checksum_accum;

void parser_feed(uint8_t byte) {
    switch (state) {
    case STATE_IDLE:
        if (byte == 0xAA) {
            state = STATE_GOT_HEADER;
        }
        break;

    case STATE_GOT_HEADER:
        if (byte == 0 || byte > sizeof(rx_buf)) {
            state = STATE_IDLE;          // invalid length — reset
            break;
        }
        expected_len     = byte;
        byte_count       = 0;
        checksum_accum   = 0;
        state = STATE_RECEIVING_DATA;
        break;

    case STATE_RECEIVING_DATA:
        rx_buf[byte_count++] = byte;
        checksum_accum ^= byte;
        if (byte_count == expected_len) {
            state = STATE_CHECKSUM;
        }
        break;

    case STATE_CHECKSUM:
        if (byte == checksum_accum) {
            packet_handler(rx_buf, expected_len);  // valid packet
        }
        state = STATE_IDLE;            // reset regardless of checksum result
        break;
    }
}
Key properties of a well-written FSM: every state has a defined response to every possible input (even if that response is "ignore and stay in this state"), and every error path returns to a known recovery state rather than getting stuck. STATE_IDLE is the recovery state here — any unexpected byte from any state can safely transition back to it.
Ring buffers (circular buffers)
The ring buffer is the standard data structure for passing data from interrupt context to main context. The interrupt handler writes bytes into it; the main loop reads them out at its own pace. Because writes and reads happen independently, the buffer decouples the two.
Head advances on write; tail advances on read. Both wrap around modulo the buffer size. Interrupt writes at head; main loop reads from tail.
// Interrupt-safe single-producer / single-consumer ring buffer
// Works without a mutex when one side is producer (ISR) and
// the other is consumer (main loop) — no simultaneous writes.

#define RING_SIZE 64   // must be a power of two for masking trick
typedef struct {
    uint8_t  buf[RING_SIZE];
    uint16_t head;     // written by producer (ISR)
    uint16_t tail;     // written by consumer (main loop)
} ring_buf_t;

static inline bool ring_push(ring_buf_t *r, uint8_t byte) {
    uint16_t next_head = (r->head + 1) & (RING_SIZE - 1);
    if (next_head == r->tail) return false;  // full — drop byte
    r->buf[r->head] = byte;
    r->head = next_head;   // publish: write head AFTER storing data
    return true;
}

static inline bool ring_pop(ring_buf_t *r, uint8_t *out) {
    if (r->head == r->tail) return false;   // empty
    *out = r->buf[r->tail];
    r->tail = (r->tail + 1) & (RING_SIZE - 1);
    return true;
}

// In the UART RX interrupt handler:
void USART0_RX_IRQHandler(void) {
    ring_push(&uart_rx_ring, USART_RxDataGet(USART0));
}

// In the main loop:
void process_uart(void) {
    uint8_t byte;
    while (ring_pop(&uart_rx_ring, &byte)) {
        parser_feed(byte);
    }
}
The ordering matters. In ring_push(), the data byte is stored before updating head. This ensures the consumer never sees an updated head pointing to a slot that does not yet contain valid data. On a single-core Cortex-M, this works without a mutex as long as only one writer and one reader exist — the write to head is a single aligned 16-bit store, which is atomic on ARMv7-M.
Driver layering
As a firmware project grows, peripheral access scattered throughout the application becomes unmaintainable. The standard fix is a three-layer model:
  • Hardware Abstraction Layer (HAL) / register layer — direct register access, vendor CMSIS/EMLIB calls. Knows about the specific MCU family. Example: USART_Tx(USART0, byte).
  • Driver layer — wraps the HAL into a functional interface that does not expose MCU-specific types to the rest of the application. Owns the ring buffer, handles interrupts, manages state. Example: uart_write(const uint8_t *data, size_t len).
  • Application layer — uses drivers through their clean API. Has no knowledge of which USART peripheral is in use, what the DMA channel number is, or how the interrupt is configured. Example: comms_send_packet(&pkt).
// Driver layer: uart.h (what the application sees)
void     uart_init(uint32_t baud);
bool     uart_write(const uint8_t *data, size_t len);
size_t   uart_read(uint8_t *buf, size_t max_len);
bool     uart_tx_busy(void);

// Driver layer: uart.c (what knows about USART0, DMA, interrupts)
// -- hides ring_buf_t, USART0, DMA channel 2, USART0_RX_IRQn
// -- application never sees any of these

// Application layer: comms.c
void comms_send_packet(const packet_t *pkt) {
    uart_write(pkt->data, pkt->len);   // no USART0 here
}
The payoff is portability and testability. To move from USART0 to USART1, you change one file (uart.c), not every call site. To test comms.c on a host machine, you provide a stub uart.c that writes to a file instead of a UART peripheral.
volatile and memory ordering
Two C keywords are essential for correct bare-metal firmware: volatile and memory barriers.
volatile tells the compiler that a variable's value may change outside the visible program flow — due to hardware, another interrupt, or DMA. Without volatile, the compiler may cache the variable in a register and never re-read it from memory, missing updates from the interrupt handler. 
// WRONG: compiler may optimise this to an infinite loop
//        because it doesn't see anything modifying 'flag'
bool flag = false;
while (!flag) { }

// CORRECT: volatile tells the compiler to re-read from memory each time
volatile bool flag = false;
while (!flag) { }

// Also required for memory-mapped peripheral registers:
// (CMSIS headers already declare these volatile — don't re-declare them)
volatile uint32_t *reg = (volatile uint32_t *)0x40000000;
*reg = 0x01;   // this write must reach the hardware, not be elided
volatile prevents the compiler from eliminating or reordering accesses to that specific variable, but it does not prevent the CPU from reordering memory operations relative to each other. On Cortex-M (which has a weakly ordered memory model for normal memory), you need memory barriers when the order of separate memory accesses matters — for example, ensuring DMA descriptor fields are committed before the DMA enable bit is written. 
// Memory barrier intrinsics (ARM CMSIS)
__DMB();  // Data Memory Barrier: all explicit memory accesses before the
          // barrier complete before any after it. Use before enabling DMA.

__DSB();  // Data Synchronization Barrier: stronger — waits for all
          // outstanding memory transactions to complete. Use before
          // modifying interrupt priority or enabling exceptions.

__ISB();  // Instruction Synchronization Barrier: flushes the pipeline.
          // Required after modifying VTOR (vector table offset) or
          // after enabling/disabling caches.

// Typical DMA setup pattern:
desc.src = source_buf;
desc.dst = dest_buf;
desc.ctrl = LDMA_CTRL_SIZE_WORD | (count - 1);
__DMB();             // ensure descriptor is written before LDMA sees it
LDMA->CHEN |= (1 << channel);  // enable — hardware reads descriptor now
