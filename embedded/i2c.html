10 · I2C
Open-Drain Bus · START/STOP · Addressing · ACK/NAK · Clock Stretching · EFR32
I2C — Inter-Integrated Circuit, often called "I-squared-C" or "I-two-C" — solves a problem SPI cannot: how do you put many devices on a bus without running a separate chip-select wire to each one? I2C does it with addressing. Every device on the bus has a unique 7-bit address baked into it at manufacture (sometimes with 1–3 address bits configurable via pins). The master initiates every transfer, calls out an address, and only the matching device responds. Two wires — SDA (data) and SCL (clock) — serve the entire bus.
The trade-off is speed. Standard mode is 100 kHz, Fast mode 400 kHz, Fast-Plus 1 MHz. This is much slower than SPI but sufficient for sensors, EEPROMs, RTCs, IO expanders, and display controllers where bandwidth is not the bottleneck. For a biosensor that produces a sample every millisecond, 400 kHz I2C is plenty.
Open-drain bus topology
Both SDA and SCL are open-drain (or open-collector) lines. No device on the bus drives the line HIGH — instead, a pull-up resistor holds the line HIGH when nothing is pulling it LOW. Any device can pull the line LOW by switching on an internal N-channel FET to ground. This means:
  • Multiple masters and slaves can share the same two wires without shoot-through — there is never a conflict between one device pushing HIGH and another pushing LOW, because nobody pushes HIGH.
  • Any device can hold the bus LOW to pause a transfer (clock stretching).
  • The bus is wired-AND: the line is HIGH only when every device releases it.
I2C open-drain bus. Pull-up resistors are the only source of logic HIGH. Multiple masters and slaves share the two wires, distinguished only by address.
Pull-up resistor sizing is a real design decision. A smaller Rp pulls the line HIGH faster (lower rise time) allowing higher speed, but draws more current whenever any device pulls LOW (I = VDD/Rp). A larger Rp saves power but limits the maximum bus speed because the RC time constant of Rp and bus capacitance sets the rise time. The I2C spec defines maximum rise time: 1000 ns for Standard mode, 300 ns for Fast mode, 120 ns for Fast-Plus.
$$R_{p,\min} = \frac{V_{DD} - V_{OL(\max)}}{I_{OL}} \approx \frac{3.3\,\text{V} - 0.4\,\text{V}}{3\,\text{mA}} \approx 970\,\Omega$$ $$R_{p,\max} = \frac{t_\text{rise}}{0.8473 \cdot C_\text{bus}} \qquad \text{(10–90\% rise time to } 0.7\,V_{DD}\text{)}$$
Typical: 2.2–4.7 kΩ for 400 kHz Fast mode on a short PCB trace
START and STOP conditions
I2C does not have a separate CS wire. Instead it uses special bus conditions to frame transactions. A START condition is defined as SDA going LOW while SCL is HIGH — this is the only time SDA is allowed to change while SCL is HIGH, so the receiver hardware can unambiguously detect it. A STOP condition is SDA going HIGH while SCL is HIGH.
START and STOP are the only moments SDA is permitted to change while SCL is HIGH. All normal data bits (D0…) transition only while SCL=LOW and are sampled on the SCL rising edge.
A Repeated START (Sr) is a START condition issued without a preceding STOP. It changes the direction of the transfer (e.g. from a write of the register address to a read of the register contents) without releasing the bus. This matters because releasing the bus between the address-write and data-read would allow another master to steal it.
Addressing and ACK/NAK
After START, the master clocks out the 7-bit slave address followed by a direction bit: R/W̄ (0 = write, 1 = read). That is the first byte on the bus. The addressed slave must pull SDA LOW on the 9th clock cycle to send an ACK. If no device recognises the address, SDA stays HIGH — that is a NAK (no acknowledge), and the master should issue a STOP or Repeated START.
The master drives 8 bits (A6–A0 + R/W), then releases SDA. On the 9th clock the slave must pull SDA LOW to ACK. If no device responds, SDA stays HIGH (NAK) and the master should issue STOP or Repeated START.
ACK after each data byte — not just the address byte — is part of the protocol. On writes (master→slave), the slave ACKs each received byte to confirm it was accepted. On reads (slave→master), the master ACKs each byte to tell the slave to keep going; on the last byte the master sends a NAK to signal it is done, followed by a STOP.
10-bit addressing extends the address space from 128 devices to 1024. The first byte sent is a special prefix (0b11110XX where XX are the top 2 bits of the address), followed by a second byte with the remaining 8 bits. 10-bit and 7-bit devices can coexist on the same bus; the prefix is reserved and will not match any 7-bit device.
Full transaction: register read
The most common I2C operation in embedded firmware is a register read: write the register address to the device, then read the data back. This requires a Repeated START between the write and read phases to avoid releasing the bus. 
// Conceptual I2C register read sequence (phase by phase):
//
// [S] [ADDR | W] [A] [REG_ADDR] [A] [Sr] [ADDR | R] [A] [DATA] [N] [P]
//
//  S  = START condition
//  Sr = Repeated START (no STOP in between)
//  A  = ACK (slave pulls LOW)
//  N  = NAK (master doesn't pull LOW — signals "last byte")
//  P  = STOP condition
//  W  = write bit (0),  R = read bit (1)
//
// Step 1: Write phase — tell the device which register to read
//   Master: S  + (device_addr << 1) | 0x00   ← ADDR | WRITE
//   Slave:  A
//   Master: reg_addr                            ← register pointer
//   Slave:  A
//
// Step 2: Read phase — switch direction without releasing the bus
//   Master: Sr + (device_addr << 1) | 0x01   ← ADDR | READ
//   Slave:  A
//   Slave:  data_byte                           ← slave drives SDA
//   Master: N                                   ← NAK = last byte
//   Master: P                                   ← STOP
If you issue a STOP between the write and read phases, the device may reset its internal register pointer. Some devices allow back-to-back reads without re-sending the register address, but many do not — always use Repeated START for multi-byte reads unless the datasheet explicitly says otherwise.
Clock stretching
Because SCL is open-drain, a slave can hold SCL LOW after the master releases it — pausing the entire transfer while it processes. The master must monitor SCL and wait until it actually goes HIGH before starting the next clock period. This is called clock stretching.
Most sensor firmware rarely needs to worry about clock stretching — the EFR32 I2C peripheral handles it in hardware. But it matters for two reasons:
  • If your master has a hard deadline and the slave stretches for too long, the transaction may time out. Always configure a bus timeout in your master driver.
  • Some slave devices use clock stretching as an analog conversion trigger signal — they stretch SCL while the ADC is running and release it when the result is ready.
EFR32 I2C API
The EFR32 has a dedicated I2C peripheral (I2C0, sometimes I2C1). EMLIB provides both a low-level register API and a higher-level I2C_TransferInit() / I2C_Transfer() state-machine driver that handles all the START, address, ACK, Repeated START, and STOP sequencing automatically. 
#include "em_i2c.h"
#include "em_gpio.h"
#include "em_cmu.h"

#define I2C_SCL_PORT   gpioPortC
#define I2C_SCL_PIN    11
#define I2C_SDA_PORT   gpioPortC
#define I2C_SDA_PIN    10

void i2c_init(void) {
    CMU_ClockEnable(cmuClock_I2C0,  true);
    CMU_ClockEnable(cmuClock_GPIO,  true);

    // Both pins open-drain, pulled high externally (or via internal pull-up)
    GPIO_PinModeSet(I2C_SCL_PORT, I2C_SCL_PIN, gpioModeWiredAndPullUp, 1);
    GPIO_PinModeSet(I2C_SDA_PORT, I2C_SDA_PIN, gpioModeWiredAndPullUp, 1);

    I2C_Init_TypeDef init = I2C_INIT_DEFAULT;
    init.enable  = true;
    init.master  = true;
    init.refFreq = 0;                     // use current clock
    init.freq    = I2C_FREQ_FAST_MAX;     // 400 kHz Fast mode
    init.clhr    = i2cClockHLRStandard;   // symmetrical SCL duty cycle
    I2C_Init(I2C0, &init);

    // Route SDA/SCL (Series 1 — check LOC for your pinout)
    I2C0->ROUTELOC0 = I2C_ROUTELOC0_SCLLOC_LOC14
                    | I2C_ROUTELOC0_SDALOC_LOC14;
    I2C0->ROUTEPEN  = I2C_ROUTEPEN_SCLPEN
                    | I2C_ROUTEPEN_SDAPEN;
}

// ── High-level register write ───────────────────────────────────
I2C_TransferReturn_TypeDef i2c_write_reg(uint8_t addr, uint8_t reg, uint8_t val) {
    uint8_t buf[2] = { reg, val };

    I2C_TransferSeq_TypeDef seq = {
        .addr  = addr << 1,              // 7-bit addr in bits [7:1]
        .flags = I2C_FLAG_WRITE,
        .buf[0].data = buf,
        .buf[0].len  = 2,
    };

    I2C_TransferInit(I2C0, &seq);
    I2C_TransferReturn_TypeDef ret;
    do { ret = I2C_Transfer(I2C0); } while (ret == i2cTransferInProgress);
    return ret;
}

// ── High-level register read (write addr, Repeated START, read) ─
I2C_TransferReturn_TypeDef i2c_read_reg(uint8_t addr, uint8_t reg, uint8_t *out) {
    I2C_TransferSeq_TypeDef seq = {
        .addr  = addr << 1,
        .flags = I2C_FLAG_WRITE_READ,     // write reg addr, Sr, read data
        .buf[0].data = ®,
        .buf[0].len  = 1,
        .buf[1].data = out,
        .buf[1].len  = 1,
    };

    I2C_TransferInit(I2C0, &seq);
    I2C_TransferReturn_TypeDef ret;
    do { ret = I2C_Transfer(I2C0); } while (ret == i2cTransferInProgress);
    return ret;
}

// Usage example: read WHO_AM_I from an IMU at address 0x6B
void check_imu(void) {
    uint8_t who_am_i;
    I2C_TransferReturn_TypeDef status = i2c_read_reg(0x6B, 0x0F, &who_am_i);
    if (status != i2cTransferDone) {
        // handle error: NAK, bus error, timeout
    }
}
I2C_FLAG_WRITE_READ is the key flag. It tells the state machine to: (1) send a WRITE address + the first buffer contents, (2) issue a Repeated START, (3) send a READ address, (4) receive into the second buffer, (5) NAK the last byte and send STOP. Without the Repeated START, many devices will reset their internal register pointer between the write and read phases.
For high-throughput or non-blocking transfers, the same I2C_TransferSeq_TypeDef can be used with the interrupt-driven driver variant: enable the I2C interrupt, call I2C_TransferInit(), and service the state machine from the I2C0_IRQHandler by calling I2C_Transfer() until it returns something other than i2cTransferInProgress.
Common problems
Problem Cause Fix
Permanent NAK Wrong address, device not powered, address pins wired incorrectly Verify address in datasheet; do an I2C bus scan (try all 128 addresses)
Bus lockup (SDA stuck LOW) Power cycle mid-transfer left a slave driving SDA LOW; master has no idea Toggle SCL 9 times to flush the slave's shift register; then issue STOP
Address conflict Two devices with the same 7-bit address on the bus Use address-configurable variants; separate onto different I2C buses
Slow rise time / data corruption Pull-up resistor too large for bus capacitance at chosen speed Reduce Rp (e.g. 2.2 kΩ → 1 kΩ); shorten traces; reduce bus speed
Intermittent NAK on read Missing Repeated START — STOP released bus, device reset register pointer Use I2C_FLAG_WRITE_READ (Repeated START); never STOP between write and read
Arbitration loss Two masters tried to start simultaneously (multi-master bus) The losing master backs off and retries; EMLIB returns i2cTransferArbLost
Bus lockup recovery. If a slave is stuck holding SDA LOW (e.g. after a power glitch interrupted a read), the standard recovery sequence is: clock SCL 9 times (enough to shift out whatever byte the slave was in the middle of transmitting), then issue a STOP condition. The EFR32 I2C peripheral has a built-in bus busy timeout — if SCL or SDA is stuck for longer than the timeout, it resets the bus and returns an error. You can also implement the 9-clock recovery manually by temporarily configuring SCL as a GPIO output.
