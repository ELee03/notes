Why DMA?
Every data transfer in a microcontroller has to move bytes from one place to another — from an ADC result register into a RAM buffer, from a RAM array out through a UART transmit register, from one region of flash to another. The naive approach makes the CPU do all of this work: read a register, store to memory, loop, repeat. For short transfers this is fine. For anything continuous — streaming sensor data, a high-speed protocol, an audio buffer — the CPU spends most of its time as an expensive bus driver and nothing else.
Direct Memory Access (DMA) solves this by giving the system a second bus master — a dedicated hardware engine that moves data autonomously. You describe the transfer once (source, destination, how many, what size), arm the DMA channel, and the CPU is free to do something else — or sleep. The DMA fires one interrupt when the job is done, and the CPU wakes up to find its buffer already full.
Transferring 8 data items from a peripheral register to RAM under three strategies.
The interrupt-driven approach is already a huge improvement over polling, but it still exacts eight context switches — eight saves and restores of R0–R3, R12, LR, PC, xPSR. For high-rate peripherals (1 kHz ADC × 8 channels, or a 921600 baud UART) those context switches accumulate. DMA eliminates them entirely, at the cost of slightly more setup.
How DMA works
A DMA controller is a second bus master that sits on the same AHB (or AHB-Lite) bus as the CPU. It can initiate read and write cycles to any address the CPU can reach — peripheral registers, SRAM, flash, external memory. The CPU configures it and then steps aside; from the bus fabric's perspective, the DMA controller looks like a second (slower) CPU.
Every DMA transfer has three ingredients:
  • Source address — where to read from. For peripheral-to-memory transfers this is a peripheral data register (e.g., ADC0→SINGLEDATA or USART0→RXDATA). For memory-to-memory it is a RAM or flash address.
  • Destination address — where to write to. Usually a RAM buffer for incoming data, or a peripheral TX register for outgoing data.
  • Transfer count — how many units to move, where a unit is 1, 2, or 4 bytes (byte / half-word / word). The controller decrements an internal counter on each beat and stops when it hits zero.
Beyond these basics, most DMA controllers add a trigger — a signal from a peripheral that says "I have data ready" (or "I'm ready to accept data"). The DMA waits for the trigger before each beat rather than running at bus speed, which naturally paces the transfer to the peripheral. A USART RX trigger fires once per received byte; an ADC trigger fires once per conversion. The DMA moves exactly as fast as the peripheral produces (or consumes) data, without the CPU doing anything.
Bus arbitration. The CPU and DMA share the same bus. When both want to access memory simultaneously, the bus arbiter gives one of them priority. Most Cortex-M implementations use round-robin or fixed-priority arbitration with "block size" — the DMA completes a configurable burst (e.g., 4 beats) before yielding the bus, ensuring the CPU still gets regular bus access and isn't completely starved.
Transfer descriptors
Older DMA controllers (like the STM32 DMA1/DMA2) store the transfer parameters directly in peripheral registers: write source address to DMA_CSxPAR, destination to DMA_CSxMAR, count to DMA_CSxNDTR, then enable the channel. Simple, but you can only describe one transfer at a time; chaining transfers means taking a done interrupt and reprogramming the peripheral registers in software.
The EFR32 uses a more capable architecture — a linked DMA (LDMA) — where the transfer parameters live in a descriptor stored in RAM. A descriptor is just a 16-byte struct (four 32-bit words):
A 16-byte LDMA descriptor. Four fields: control word, source address, destination address, and link word.
The power of this layout is in the LINK field. When the DMA hardware finishes a descriptor, it checks the LINK word. If it is non-zero, it fetches the address stored there and loads a new descriptor from RAM — no CPU involvement. This is what makes it a linked DMA. You can build an entire chain of transfers that execute back-to-back without a single interrupt or CPU instruction. If the last descriptor in the chain points back to the first, the chain runs forever — which is exactly the circular buffer pattern.
CTRL field Meaning Common values
xferCnt Transfer count minus one (11-bit field → up to 2048 beats) N − 1
size Unit size per beat 0=byte, 1=half-word, 2=word
srcInc / dstInc How much to advance the address after each beat. Set to 0 for peripheral registers (same address every beat); set to +1/+2/+4 for memory (advance through buffer). 3=no-inc, 0=+1unit
blockSize Beats per bus arbitration slot. Higher values give DMA more burst bandwidth but reduce CPU bus access. 1 (most common)
doneIfs Raise an interrupt when this descriptor completes. Can be set on some descriptors but not others in a chain. 0 or 1
structType Type of descriptor action: xfer (normal data move), sync (wait for a software sync bit), write (write a constant to memory). 0=xfer
The EFR32 LDMA
The EFR32 LDMA (Linked DMA) provides 8 independent channels. Each channel runs its own descriptor chain and can be triggered by a different peripheral signal — you could simultaneously stream ADC data on channel 0, transmit a UART packet on channel 1, and copy a lookup table in memory on channel 2.
Each channel has its own hardware registers:
  • CH[n].REQSEL — selects which peripheral signal triggers this channel. There are up to 64 sources (USART RX, USART TX, ADC scan, ADC single, TIMER overflow, etc.).
  • CH[n].CFG — arbitration configuration: number of bus slots this channel gets per arbitration round.
  • CH[n].LOOP — loop counter for LOOP-type descriptors (rarely used; LINK chaining is more flexible).
  • CH[n].CTRL / SRC / DST / LINK — read-only mirrors of the currently-executing descriptor (useful for debugging).
You rarely write to these registers directly. Silicon Labs provides the EMLIB LDMA driver (em_ldma.h), which wraps everything in two structs and a handful of macros:
  • LDMA_TransferCfg_t — channel-level configuration: which peripheral trigger to use.
  • LDMA_Descriptor_t — the 16-byte descriptor struct described above.
  • LDMA_StartTransfer(ch, &cfg, &desc) — loads the first descriptor into channel ch and arms it.
Series 2 difference: LDMAXBAR. On EFR32 Series 2 devices (Cortex-M33), peripheral DMA signals are not routed directly through the LDMA. Instead they go through an external arbiter called LDMAXBAR (LDMA Crossbar). Each LDMA channel selects its signal from LDMAXBAR (up to 32 signal slots). The EMLIB API handles this transparently — you still pass a ldmaPeripheralSignal_xxx constant to the transfer config and EMLIB programs both LDMAXBAR and LDMA for you.
Code patterns
Three patterns cover the vast majority of DMA use cases in embedded firmware: single shot transfers, continuous peripheral streaming, and memory-to-memory copies. All three use the same LDMA_StartTransfer() call — the difference is entirely in how the descriptor (or descriptor chain) is constructed.
Pattern 1 — Single transfer (ADC one-shot)
The simplest case: take N samples from the ADC scan result register and fill a RAM buffer. One descriptor, no link. The channel fires an interrupt when done and stops. 
#include "em_ldma.h"
#include "em_adc.h"

#define NUM_SAMPLES  256
static uint32_t adc_buf[NUM_SAMPLES];

void dma_adc_single_shot(void) {
    // Descriptor: move NUM_SAMPLES words from ADC0->SCANDATA into adc_buf.
    // SINGLE means no chaining — one descriptor, one interrupt, done.
    LDMA_Descriptor_t desc = LDMA_DESCRIPTOR_SINGLE_P2M_WORD(
        &ADC0->SCANDATA,   // src: peripheral register (no src increment)
        adc_buf,           // dst: RAM array (auto-incremented each beat)
        NUM_SAMPLES        // xferCnt = NUM_SAMPLES - 1 internally
    );

    // Transfer config: triggered by ADC0 scan-done signal
    LDMA_TransferCfg_t cfg = LDMA_TRANSFER_CFG_PERIPHERAL(
        ldmaPeripheralSignal_ADC0_SCAN
    );

    LDMA_Init_t init = LDMA_INIT_DEFAULT;
    LDMA_Init(&init);

    // Start on channel 0 — fires LDMA_IRQn when complete
    LDMA_StartTransfer(0, &cfg, &desc);
}

void LDMA_IRQHandler(void) {
    uint32_t flags = LDMA_IntGet();
    LDMA_IntClear(flags);

    if (flags & (1u << 0)) {
        // Channel 0 done — adc_buf now holds NUM_SAMPLES results
        process_samples(adc_buf, NUM_SAMPLES);
    }
}
Notice LDMA_DESCRIPTOR_SINGLE_P2M_WORD is a macro that fills in the CTRL word correctly: no source increment (the ADC data register address is fixed), destination increment of +4 bytes (word = 4 bytes), unit size = word, no LINK address (single shot).
Pattern 2 — UART TX without blocking
DMA-driven UART transmit is the memory-to-peripheral mirror of the above. The USART TX-buffer-level signal (TXBL) fires whenever the transmit FIFO has room; the DMA feeds it one byte at a time. 
static const uint8_t tx_message[] = "Hello from DMA\r\n";

void dma_uart_transmit(void) {
    LDMA_Descriptor_t desc = LDMA_DESCRIPTOR_SINGLE_M2P_BYTE(
        tx_message,         // src: RAM (auto-incremented)
        &USART0->TXDATA,    // dst: USART TX register (no dst increment)
        sizeof(tx_message) - 1
    );

    LDMA_TransferCfg_t cfg = LDMA_TRANSFER_CFG_PERIPHERAL(
        ldmaPeripheralSignal_USART0_TXBL
    );

    LDMA_StartTransfer(1, &cfg, &desc);
    // CPU is now free — bytes drip out of USART0 at baud rate
    // LDMA_IRQn fires when all bytes are sent
}
Continuous streaming: the ping-pong buffer
A single-shot transfer works for sporadic data. For continuous data — streaming ADC samples at 1 kHz indefinitely — you need a way to run DMA forever while the CPU processes what has already arrived. This is the ping-pong (or double-buffer) pattern.
Split the sample buffer into two halves. Build two descriptors: the first fills the ping half and links to the second; the second fills the pong half and links back to the first. Both have doneIfs=1 so each completion fires an interrupt. The result is a perpetually-running loop: while DMA is filling pong, the CPU is processing ping, and vice versa.
Two linked descriptors alternate between ping and pong buffers. Each completion fires an IRQ; the CPU processes one half while DMA fills the other.
#define HALF  128
static uint32_t ping_buf[HALF];
static uint32_t pong_buf[HALF];

// Two descriptors stored in RAM — must be 4-byte aligned
static LDMA_Descriptor_t desc[2];

void dma_adc_stream_init(void) {
    // desc[0]: fill ping_buf, then load desc[1]
    desc[0] = (LDMA_Descriptor_t)LDMA_DESCRIPTOR_LINKREL_P2M_WORD(
        &ADC0->SCANDATA,  // src
        ping_buf,         // dst
        HALF,             // count
        +1                // link offset: +1 descriptor forward = desc[1]
    );
    desc[0].xfer.doneIfs = 1;  // interrupt when ping_buf full

    // desc[1]: fill pong_buf, then link back to desc[0]
    desc[1] = (LDMA_Descriptor_t)LDMA_DESCRIPTOR_LINKREL_P2M_WORD(
        &ADC0->SCANDATA,
        pong_buf,
        HALF,
        -1                // link offset: -1 descriptor back = desc[0]
    );
    desc[1].xfer.doneIfs = 1;  // interrupt when pong_buf full

    LDMA_TransferCfg_t cfg = LDMA_TRANSFER_CFG_PERIPHERAL(
        ldmaPeripheralSignal_ADC0_SCAN);

    LDMA_StartTransfer(0, &cfg, &desc[0]);
}

void LDMA_IRQHandler(void) {
    uint32_t flags = LDMA_IntGet();
    LDMA_IntClear(flags);

    if (flags & (1u << 0)) {
        // Determine which half DMA just finished by inspecting
        // the current destination pointer
        uint32_t next_dst = LDMA->CH[0].DST;
        if (next_dst == (uint32_t)pong_buf) {
            // DMA is now filling pong — safe to process ping
            process_samples(ping_buf, HALF);
        } else {
            // DMA is now filling ping — safe to process pong
            process_samples(pong_buf, HALF);
        }
    }
}
Descriptor alignment. LDMA descriptor arrays must be declared as static (or global) and be naturally aligned — the hardware reads them via the 4-byte-aligned LINK address. Never put a descriptor on the stack inside a function that returns; the DMA hardware will keep reading from that stack address long after the frame is gone.
What about LINKREL vs LINKABS? The relative form (LINKREL) stores an offset in units of 16 bytes (one descriptor) from the current descriptor's address. This is convenient when descriptors are in a contiguous array — +1 means "the next element," -1 means "the previous." The absolute form (LINKABS) stores the physical RAM address directly. Use absolute links when descriptors are scattered in memory or when the chain is more complex than a simple array.
Memory-to-memory copy
DMA can also move data between two RAM regions — useful for initializing large buffers, copying data into a second structure for processing, or preparing a DMA-ready ping-pong scratch buffer. There is no peripheral trigger; the transfer uses a software (immediate) trigger and runs as fast as the bus allows. 
void dma_memcpy(void *dst, const void *src, size_t words) {
    LDMA_Descriptor_t desc = LDMA_DESCRIPTOR_SINGLE_M2M_WORD(src, dst, words);
    LDMA_TransferCfg_t cfg = LDMA_TRANSFER_CFG_MEMORY();
    LDMA_StartTransfer(2, &cfg, &desc);
    // For a blocking call, wait until done:
    while (LDMA_TransferDone(2)) { }
}
What DMA can't do
DMA excels at bulk data movement but is the wrong tool for some jobs:
  • Conditional transfers. The DMA controller has no ALU. It cannot check a value, branch, or apply logic. If you need "copy bytes until you see a 0x0A," you need the CPU — or you use DMA to fill a buffer and then scan it in software.
  • Non-contiguous peripheral access. DMA moves data in linear address sequences. Reading alternating registers or striding through a struct requires either a CPU loop or a scatter-gather extension (which the EFR32 LDMA supports via write descriptors — but this quickly becomes complex).
  • Very short or very sporadic transfers. The overhead of setting up a descriptor (a few dozen cycles) outweighs the benefit for 1–4 byte transfers that happen rarely. A plain interrupt or even polling wins there.
  • Cache coherency (EFR32 Series 2 with M33). On devices with an instruction cache, the cache and DMA see the same physical memory. If DMA writes to a region that the CPU has cached, the CPU will read stale data. The EFR32 Series 2 does not have a data cache (only an instruction cache), so this is not a practical concern for data buffers — but it is critical on Cortex-M7 and Cortex-A devices.
Within those boundaries, DMA is indispensable for any embedded system that moves significant data. A biosensor front-end sampling 8 channels at 1 kHz (8000 samples/second, 32 bytes each = 256 kB/s) would need the CPU to handle 256,000 interrupt entries per second without DMA. With DMA and a ping-pong buffer, the CPU sees 2 × (rate / HALF) = at most a handful of IRQs per second and is otherwise completely free.
